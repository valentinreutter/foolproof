Foolproof

The manual for working with an AI agent when you cannot read code.

Never trust what you cannot see.

April 2026 · A living document

This book is written for the person who is building real software with AI agents without being able to read the code those agents produce. It names the one principle that keeps such work safe — never trusting what you cannot see — and then shows, concretely, how to compose a stack, set up a machine, run sessions, and recover from the specific failures that non-coders hit.

How to read it

Once straight through for the shape. Then as reference: the tier ranking in Chapter 4, the seven-day setup in Chapter 5, the prompt library in Chapter 9, the playbooks in Chapter 13.

Parts I and II are theory. An hour. The principle and the tier framework that ranks tools against it. Part III is the build — one week, two hours a day. Part IV is the craft of working with agents. Parts V and VI are daily operation and resilience. The appendix is reference.

Specific services are named as worked examples for a solo SaaS founder in Germany, April 2026. The structure is universal; the principle is evergreen; the tools are timestamped and marked as such. Substitute by the framework in Chapter 4; the principles don't move.

Who this is for

If you cannot read code fluently, this book is written for you. If you can, the early chapters will feel slow — skim to Chapter 4. The arc runs from accessible to dense.

Part I

The principle

Three chapters. The one idea that makes everything else make sense, the two ways it fails, and what it does not claim to do.

Chapter 1Never trust what you cannot see

You cannot read the code your agent writes. Not with fluency. That's the constraint everything else in this book works around.

The naive answer is trust. Trust the agent. Trust the model. Trust the tests if they pass. That answer breaks the first time something silent goes wrong — the migration that ran but skipped a column, the webhook that fired but never arrived, the dependency that installed and quietly phoned home. Trust scales badly when you cannot read the thing you're trusting.

The working answer is a single rule, applied ruthlessly: never trust what you cannot see.

Every capability in your system must be visible to you through a surface you can open — a dashboard, a canvas, a preview, a file tree. Agents can act on that same state. Both sides see the same truth.

Two surfaces, one state. The dashboard you open and the interface the agent calls are projections of the same thing. Change one, the other sees it. When they drift, or when a capability exists only on one side, the system has failed even if nothing has broken yet.

This property — the technical name is correspondence — is what lets a non-coder verify by looking instead of by reading. It is the structural precondition for everything else in the book.

A concrete picture

Without it. An operator runs a SaaS on a €20/month VPS. The agent has root. Backups are cron jobs. One morning a user reports charges that didn't go through. The operator asks the agent; the agent reports that the payment webhook handler crashed at 03:14 UTC and has been restarted. The operator has no independent view. No dashboard. No event log. No replay. The agent's report is the only record.

With it. Same operator, different stack. Payments through Stripe — dashboard, API, MCP all hitting the same event stream. Database in Supabase — Studio, SQL, MCP all one project. When the same failure happens, the operator sees it in Stripe's Events view at 03:15, clicks the failed webhook, expands the payload, hits replay — all without reading code. The agent can do the same replay via MCP while the operator watches. Two layers agree on what happened; recovery is a click or a command, and both surfaces confirm it succeeded.

The difference isn't that the second setup is safer in the abstract. It's that failure is visible to the operator and recovery is actionable from either side.

Where the rule works, and where it doesn't

The rule applies cleanly to resources — the things your system holds. A database row, a user account, an environment variable, a Stripe product, a DNS record, a deployed build. Every resource can have a visible surface and an agent interface pointing at the same state.

The rule does not apply cleanly to behavior — the things your system does. Your frontend code and the rendered page are not two views of one state; they're connected by a build pipeline. For behavior — which for a non-coder means "the frontend" and "custom business logic" — the best available isn't correspondence but verification by observation: click the preview, try the feature, read the Playwright screenshot. These are compensations for missing correspondence, not instances of it.

Know which side of the line your product sits on. A backend-heavy SaaS with a thin frontend gets near-full correspondence coverage. A frontend-heavy product with complex custom UI gets weaker coverage, and the non-coder path is genuinely harder there.

Chapter 2Underlap, overlap, bridges

Correspondence is a property of a single tool. Composition — what happens when you chain tools together — is where real systems fail.

The two failures named precisely

Underlap is a capability that exists in only one layer.

  • Agent-side: the capability is visible but not agent-driveable. A Notion database the MCP can read but not fully mutate. A Webflow site you can drag but not script.
  • Visible-side: the capability is agent-driveable but not visible. A bare VPS with SSH only. A headless CMS with complete API and no admin UI. The agent can act; the operator cannot verify.

Overlap with drift: both layers exist but fail to stay synchronized. A Stripe price changes in code; the dashboard label wasn't updated. .env.local on the laptop, .env on staging, and Vercel's environment UI each hold different values for the same key. A webhook URL points at an endpoint the main app moved months ago.

Overlap is worse than underlap because it produces false visibility. You open the dashboard, see a value, act on it — but the real state is different. The loop between looking and acting is broken without any error appearing.

Every boundary is a correspondence

A solo operator is not running one tool. They're running a dozen wired together: domain, DNS, edge function, app server, database, jobs, email, analytics, errors, billing. Each arrow is a bridge. Each bridge is a place where correspondence must be maintained by you, because no vendor owns both sides.

The minimum-correspondence principle

The number of correspondences you must maintain scales with the number of tool boundaries. Prefer fewer tools with tight internal correspondence over many tools each with partial correspondence. If you can collapse two tools into one without losing correspondence, do it.

Checklist before adding a tool

  • Does it have a GUI a non-coder can open and use?
  • Does it have an agent interface (MCP, API, CLI) that reaches the same state?
  • Do changes in one surface appear in the other within a known time window?
  • If it's a bridge to another service, does it pass the three-trait test (Chapter 12)?
  • What does adding it take your total correspondence count to? Is the capability gain worth the increase?

Chapter 3What this book does not do

A book that promises foolproofness is lying. This one does not. Before applying any recommendation, know what the principle does not buy you.

  • It does not prevent compromise. Phishing that tricks you into authorizing a malicious MCP server bypasses every tier ranking. Supply-chain worms in dependencies run regardless of container boundaries.
  • It does not eliminate silent bridge failure. Even top-tier tools have bridges to lower-tier systems somewhere. The three-trait test catches most cases, not all.
  • It does not self-verify agent output. Code can pass every test and be wrong in production. Correspondence makes actions visible; it does not make judgment correct.
  • It does not extend to frontend code for a non-coder. The rule applies to resources, not to behavior produced by code you cannot read. Name the gap; don't pretend it isn't one.
  • It does not remove human judgment. Every verify gate requires you to look at something and decide. The rule reduces what you must decide; it does not replace the deciding.
  • It does not make rated tools infallible. Supabase can have an outage. Vercel can fat-finger a config. The tier ranking is about how detectable and recoverable failures are, not whether they happen.
  • It does not replace legal, financial, or tax advice. Appendix B is context for conversations with professionals, not a substitute.

Where the principle doesn't apply at all

The principle is for problem-shaped work — questions with discoverable answers. It does not apply to emotional processing, creative generation for its own sake, meandering conversation, embodied attention, personal journaling, or relationships. When the point is the process, the feeling, the practice, or the connection, the agent does not belong. Put it down. The diagnostic: is there an unknown here — a definite thing I'm trying to produce, learn, decide, or verify? If yes, the agent fits. If no, keep it out.

What the principle does do

  • Makes failures detectable by a non-coder at near-zero latency.
  • Makes recovery actionable from either side.
  • Makes decisions traceable — same event, same timestamp, both surfaces.
  • Reduces the footprint of "trust me" to a known, bounded, auditable area. Not zero. Bounded.

Part II

The stack

One chapter. A framework that ranks every category of tool by how well it preserves the principle, a specific composition as of April 2026, and the test for any tool that ships tomorrow.

Chapter 4Tiers and composition

Timestamped

This ranking reflects April 2026. The tiers are evergreen; tool assignments are a snapshot. Re-verify before major decisions; run the evaluation test at the end of this chapter whenever a new tool appears.

The tiers

Tier S — near-perfect correspondence. Remote MCP with OAuth, same state as dashboard, agent does ~95%+ of what the GUI does, changes appear within seconds either direction.

Tier A — strong, named gaps. Remote MCP or complete API, most parity, a small set of GUI-only tasks (org settings, destructive actions, inherently visual work like session replay). Gaps documented.

Tier B — partial, bridge-dependent. API the agent can reach, no first-party MCP or only a slice. Composition requires writing your own bridge.

Tier C — broken divergence. GUI and API have diverged enough that they disagree in operation. Notion, Jira, old-guard SaaS. Community MCP servers drift from vendor changes within weeks.

Tier D — one-sided, usually dangerous. Either GUI-only with no agent interface (most classic no-code), or agent-first with opaque state. In 2025 this category included all the "AI-native app builders" — Bolt, Lovable, v0, Replit Agent. In 2026 the category has split, which matters; see below.

The 2026 split: AI-native builders that are now Tier A/B vs those still Tier D

The AI-native builder category has bifurcated. Tools that went one way grew real correspondence; tools that went the other doubled down on opacity.

Grew correspondence (now Tier A or B, depending on your workflow): Lovable. Native Supabase MCP integration — the agent operates through Supabase, not through a hidden backend. Bidirectional GitHub sync — generated code is standard React + Supabase and lives in a repo you control. Personal Connectors for any MCP server — your own memory vault, your own tools. Visual canvas, chat, preview URL, Supabase Studio — four visible surfaces pointing at the same state. SOC 2 + ISO 27001 + GDPR. The agent's build process is still somewhat opaque, which keeps it from Tier S, but the generated output is fully portable and the state is all in Supabase. For a non-coder operator this is the first AI-native builder that clears the bar.

Still Tier D: Bolt.new with Bolt Cloud. Replit Agent. Mocha. Anything where the backend is proprietary and non-portable, or the build process is entirely opaque and unsynced to an inspectable repo. The test: if the vendor disappeared tonight, could you take your work and continue elsewhere within a week? Tier D fails this test; current-tier-A AI builders pass it.

How to evaluate a new tool

  1. Does it have a visible surface a non-coder can open?
  2. Does it have an agent interface reaching the same state? MCP beats OpenAPI beats scripted API beats scraped UI.
  3. Do the surfaces agree? Change one; verify the other within 60 seconds.
  4. What's the bridge count to your existing stack after adoption?
  5. If the vendor disappeared tonight, could you leave within a week with your work intact?

Pass all five: adopt. Fail any: substitute, wait, or accept as a documented Tier-B-or-worse dependency. When in doubt, don't adopt. Each tool is correspondence work; decayed correspondence is the silent failure this book is structured to prevent.

Setup

One chapter, branched by path. Path A is seven days. Path B is four days. Both end with a verification sweep that either closes the setup or names the gap.

Chapter 5The setup

Two hours a day, focused. Hardware keys arrive around Day 2 so most people need to order them. Backup is anchored Day 3 or 4 because when your setup fails, backup is what recovers it. The last day is verification — no new work, only checking.

Meta-pattern

Write the setup as a markdown document where each step is both human-readable and executable. The document is the documentation. Ask Claude Code (Path A) or a plain-English chat (Path B) to run each step in plan mode — you review, it executes, the output is the receipt.

Before you start (both paths)

A Mac running macOS. A YubiKey 5C NFC ordered with a backup key — one for daily, one for a sealed envelope. A credit card. About two hours a day for the duration. If any of those are missing, pause here.

Day 1 — Identity (both paths) · 90 min

Goal: email on a domain you own, protected by a password manager with a memorable master password. By end of day: Porkbun + custom domain + paid email provider sending from it + Bitwarden + FileVault on.

  1. Buy the domain at Porkbun (10 min). 2FA on. yourname.com or .xyz — this is infrastructure, not branding. Auto-renew on.
  2. Decide your DNS path (15 min). Path A keep DNS at Porkbun and use Porkbun email forwarding; simpler, fewer accounts. Path B move DNS to Cloudflare and use Email Routing; best practice, requires a Cloudflare account, one DNS propagation wait.
  3. Create your main forward (10 min). Use hello@ — generic, professional, privacy-friendly. Forward to your existing inbox.
  4. Test receiving (5 min). From a different account send a short email to hello@yourdomain. It should arrive within 10 minutes. If not, diagnose now.
  5. Upgrade to a paid email tier (20 min). Required for custom-domain sending. Fastmail €5/mo, Proton Mail Plus €4/mo, or equivalent. Add your domain in the provider's settings, add the sending identity, click the verification email link, set the custom address as default sender.
  6. Plan your aliases (10 min): hello@ main · ai@ AI services · cloud@ infrastructure · code@ repos · billing@ invoices · newsletter@ disposable. All forward to the same inbox. Separation is for filtering and damage limitation.
  7. Bitwarden, FileVault, SIM PIN (15 min). Strong master password (diceware). Print emergency sheet; envelope. FileVault on; recovery key in envelope. Call mobile carrier, ask for port-out PIN to prevent SIM-swap.
Blindspot closed today

Portable identity. Your email is on a domain you own. If your provider suspends you, change the forward and keep working. SIM PIN prevents the most common 2FA bypass.

Day 2 — Hardware keys (both paths) · 90 min

Goal: every critical account protected by a physical key. YubiKeys arrive today.

Order of enrollment: email first (can reset everything else) · password manager second (protects the rest) · AI accounts, memory vault, code hosts, cloud providers third · other services over the week.

For each service: enroll both primary and backup. Do not enroll only one. The pattern is the same everywhere — Security settings → Add security key → tap primary → tap backup → print recovery codes → date and service on the printout → envelope → disable SMS 2FA on that service.

Enforcement, not willpower

Track enrollment in a checklist: one row per service; columns for primary-key, backup-key, TOTP backup, SMS-disabled, codes-printed. A row with SMS-disabled ticked and backup-key empty is a lockout waiting to happen. Find and fix.

End of day. Envelope contains FileVault recovery key, Bitwarden emergency sheet, backup YubiKey, recovery-code printouts. Seal. Date. Store somewhere safe but accessible.

Day 3 — Mac substrate · 90 min

Path A: install Claude Desktop + Claude Code (verify plan mode with Shift+Tab twice) · OrbStack (GUI matches orb list) · Little Snitch in Alert mode · primary browser with Work/Personal/Banking profiles · Bitwarden CLI · Infisical CLI.

Path B: install Claude Desktop · primary browser with Work/Personal/Banking profiles · Bitwarden CLI. You don't need OrbStack, Little Snitch as load-bearing, or Infisical at this stage — the build agent runs on Lovable's servers, not yours. Little Snitch stays nice-to-have for general Mac hygiene; not required.

Day 4 — Backup (both paths) · 120 min + wait

Goal: 3-2-1-1-0. Three copies, two media, one offsite, one immutable, zero errors verified.

  1. External SSD + Time Machine. First backup runs. Verify via GUI and tmutil latestbackup — same timestamp.
  2. Backblaze B2 account. Bucket with Object Lock on, Compliance mode, 30-day retention.
  3. Application Keys scoped to the bucket. Save both parts in Bitwarden as one entry — the application key shows only once.
  4. Arq 7. Two backup sets: ~/Documents + ~/Code → B2 · ~/Library/Application Support/Anthropic + Basic Memory vault → separate.
  5. First backup to B2. Wait for completion.
  6. Verified restore — not optional. Pick a random file from yesterday. Delete local. Restore from B2 via Arq. Open. Confirm identical. Delete again. Restore from Time Machine. Confirm identical.
Footgun: retention period

Pick 30 days on your first bucket, not a year. Compliance mode means nothing can delete before retention expires — not you, not support, not an emergency. People who set 1-year retention on their first bucket have ended up paying for an accidental 500 GB upload for 12 months with no way to stop the bleeding. 30 days defeats ransomware and agent accidents; long enough to be safe, short enough to limit self-damage. If you later want longer retention, create a second bucket. Do not start with windows you cannot back out of.

Weekly restore test

Calendar event every week: "Backup verified restore." Note with one row per week: date, file restored, source, result. Empty cell at week end = backup not known to be working. This is the "zero errors verified" line of 3-2-1-1-0.

Day 5 — Stack and MCP · 120 min

Path A — connect Claude Code to each tool:

  • Supabase. Add mcp.supabase.com via OAuth.
  • Vercel. Add mcp.vercel.com.
  • Cloudflare. Transfer DNS from Porkbun. Add mcp.cloudflare.com.
  • Stripe. Test mode. Add mcp.stripe.com.
  • Linear. Native MCP.
  • Playwright MCP. claude mcp add playwright npx @playwright/mcp@latest.
  • Basic Memory Cloud. Add https://cloud.basicmemory.com/mcp via OAuth.

Open each dashboard. Run one read-only query via each MCP. Confirm both surfaces see the same resources.

Path B — collapse into Lovable:

  • Supabase. Create the project. Schema work will happen through Lovable's agent.
  • Lovable. Sign up. Connect Supabase in project settings (reads your existing schema).
  • GitHub. Authorize Lovable to sync code to a repo. You never open this.
  • Basic Memory Cloud. In Claude Desktop, add https://cloud.basicmemory.com/mcp via OAuth. In Lovable, Settings → Connectors → Personal connectors → New MCP server → paste the same URL. Both agents now read the same vault.
  • Cloudflare. DNS. Point your custom domain at Lovable when ready.

Open Lovable's canvas. Open Supabase Studio. Prompt Lovable to make a tiny change. Verify the change appears in Studio within seconds. That's your correspondence check.

Day 6 — Observability and secrets · 90 min

Both paths: Sentry (SDK configuration ready for when app exists) · PostHog · Better Stack (first uptime monitor). Postmark or Resend for transactional email, SPF + DKIM configured. Granola installed and test-recorded.

Path A only: Infisical project with a test secret; verify dashboard and infisical secrets CLI show the same value.

Both paths — honest deferral: these observability tools belong in the book, but if you're pre-launch with zero users, they can be installed without the full integration work. Configure SDKs enough to be ready; wire up rich dashboards when real traffic exists.

Day 7 — Verification sweep (both paths) · 120 min — no new work

Goal: confirm the whole stack corresponds. Nothing new gets added.

Open your Basic Memory vault. New note: "Day 7 sweep" with today's date. Tick each item when confirmed:

  • Every support-capable service has primary and backup YubiKey enrolled.
  • Every service has recovery codes printed and sealed.
  • Bitwarden vault accessible on Mac and phone.
  • FileVault on; recovery key in envelope.
  • Time Machine backup within last 24h.
  • B2 via Arq backup within last 24h.
  • One random file restored from each backup; contents verified; Arq log shows zero errors.
  • (Path A) Claude Desktop shows every MCP connection green; one read-only query succeeds via each.
  • (Path B) Lovable canvas opens; Supabase Studio opens; prompting a tiny change in Lovable appears in Studio within 60s; Basic Memory reads from both Claude Desktop and Lovable.

Part IV

Working with agents

Four chapters. The craft of working with an agent in a way that preserves the principle. Pólya's shape; four durable invariants; how memory works; the prompts you'll actually use.

Chapter 6Pólya and the discipline of understanding

The deepest skill in working with an AI agent is not prompt craft. It is understanding what you are actually trying to do before asking for help. Pólya named the shape in 1945. Nothing has changed since.

Pólya's four phases

  1. Understand. Name the unknown (what are we producing?). Identify the data (what's given?). State the condition (what must be true?). Is the condition sufficient to determine the unknown? Insufficient? Contradictory?
  2. Devise. Have you seen this before in another form? Can you restate it? Can you solve a simpler version first?
  3. Carry out. Execute each step. Check each step. Can you see clearly that it's correct?
  4. Look back. Can you check the result? Derive it differently? Use the method for another problem?

John Conway, in the 2004 foreword to Pólya's book, names the deepest point: often the hardest part of researching a problem is understanding precisely what that problem says. Experienced practitioners spend more time on Phase 1 than beginners think is reasonable. Beginners skip to Phase 3 and solve the wrong problem.

Why this is central to agents

The number one failure mode is the agent solving the wrong problem. You gave a vague instruction; the agent filled in the blanks with plausible guesses; it produced beautiful work addressing a problem you don't have. The cure is Pólya Phase 1, applied to your own prompts.

Pólya's three questions, for agents

Unknown: what specifically am I asking the agent to produce? One sentence.
Data: what's already in place that the agent needs to know — stack, current state, prior decisions?
Condition: what must be true of the output for it to count as success? What must not be true? What are the non-negotiables?

If the condition is insufficient — if multiple acceptable outputs all satisfy it — the agent will pick one, and you'll be surprised at which. Refine until it points.

When you find yourself iterating many turns on what feels simple, stop. You skipped Phase 1. Write the unknown, data, condition, restart with them embedded in the first message. The work is usually one turn after that.

The phases applied live

Devise is where plan mode in Claude Code or the plan-first prompt in Lovable earn their place. The agent proposes; you read; you approve or revise before execution.

Carry out is verify gates and receipts. Each step produces evidence — build log, screenshot, SQL result. "Done" without evidence is a format error.

Look back is the phase most skipped and where compounding lives. At end of every substantive session: Can I check the result? Could I have done this differently? Can I use the method elsewhere? The output of Phase 4 goes into your memory vault. A decision becomes an observation. A reversed approach becomes a [reversed] entry future sessions will see. Skip it repeatedly and each session is isolated. Do it even briefly and each session makes the next one better.

Chapter 7The four durable invariants

Prompts date. Models change. The invariants that make prompts work don't. Four principles that outlasted 2023 and will outlast 2029.

Invariant 1 — Verification must cost less than the action it verifies

If checking whether the agent did X right takes more time than doing X yourself, you'll stop checking. The whole system collapses. Every workflow, pattern, and tool choice in this book exists to drive verification cost down — receipts cheaper to skim than re-derive, screenshots cheaper to look at than reproduce, state files cheaper to re-read than rebuild. When verification becomes expensive, the system is failing. Redesign until it's cheap again.

Invariant 2 — Agents drift proportional to session length

Every instruction you set at the start erodes as the session grows. By turn 15 or 20 the agent treats early constraints as background noise. This is not a bug; it's a structural property of how models use context. Operational consequence: for long tasks, start fresh sessions often, each with relevant state reloaded, rather than one mega-session. If you notice the agent violating a rule you set at turn 3, don't remind — reload.

Invariant 3 — Raw output is verification; summaries are attestations

When the agent says "tests passed," that's an attestation you must trust. When it pastes the last 20 lines of npm test output with "12 passed, 0 failed," that's verification. Agents will attest to work they didn't do. They cannot paste raw output they didn't get. Always ask for raw output when stakes are real.

Invariant 4 — Context is a budget, not free

What the agent "knows" in a session is exactly what's in its context window. Loading a 50-page document costs tokens that can't be spent on reasoning. Irrelevant context is worse than none — it distracts. Context engineering is deciding what to load at session start, what to load on demand, what to never load, and what to synthesize from raw material into smaller canonical files.

Context engineering, concretely

Four tiers, ranked by how often you spend them:

  • Always loaded. Your cross-project standing-orders note in Basic Memory. For code sessions, the repo's AGENTS.md or CLAUDE.md. Under 1000 words each.
  • Loaded per project. The project's state file in Basic Memory. 500–800 words. Architecture, decisions, reversed approaches, open questions.
  • Loaded on demand. Specific docs, past retrieval, files you point the agent at. Current task only.
  • Never loaded. Raw transcripts of old conversations. Logs of months-old work. Anything already synthesized — you want the synthesis, not the raw material.

The single most important habit is synthesis over import: produce small canonical state files from scattered past thinking rather than dumping old conversations in. If you find yourself pasting long documents repeatedly, your state files are wrong. Fix them; stop pasting.

Chapter 8Memory, agents, and the rules of the repo

Three artifacts do the daily work. Your standing orders in Basic Memory tell any agent who you are. Your project state files tell them what this specific project is. Your AGENTS.md or CLAUDE.md at a repo root (Path A) or Lovable project instructions (Path B) tell them the rules for this specific build. All three compose; none replaces the others.

Basic Memory, used correctly

Basic Memory is a semantic knowledge graph, not a folder of notes. Each note has three sections the graph understands:

  • Frontmatter — title, tags, permalink. Stable identifier.
  • Observations — categorized atomic facts, each indexed individually. Format: - [category] the fact #optional-tag. Categories: [decision], [fact], [technique], [requirement], [insight], [reversed], [rejected], [open]. Use the full set — they encode not just what's true but what's not true anymore.
  • Relations — typed links. Format: - implements [[Other Note]]. Types: implements, requires, part_of, extends, contrasts_with, depends_on, affects. Forward references resolve when targets are created.
The mutability split — critical

Basic Memory stores stated state. It does not observe live state. Writing current-schema facts to Basic Memory means your vault will drift the moment Supabase changes. The split:

Belongs in Basic Memory (immutable intent): product decisions with rationale, rejected paths with reasons, reversed approaches, design philosophy, vision, competitive frame. Things that are opinions and history.

Does not belong in Basic Memory (mutable facts): current table count, current deployment ID, current open bugs, current environment variables. These you derive live at session start via Supabase MCP, Vercel MCP, Sentry MCP, or Lovable's canvas. Never cache a fact in Basic Memory that a live MCP can answer faster.

The state file pattern

One per project. Path: [project-slug]/state.md. Structure:

---
title: Project — State
tags: [project, state]
---

# Project

## Context
One paragraph in plain language. What, for whom, why.

## Observations
- [decision] Choices made, dated
- [requirement] Things that must be true
- [insight] Things learned that generalize
- [reversed] Decisions no longer held, with reason — CRITICAL
- [rejected] Approaches refused, with reason
- [open] Genuinely unresolved

## Relations
- implements [[Other Note]]
- requires [[Other Note]]
- part_of [[Parent]]

## Next concrete step
NOT a list — the single thing I'd do next with two hours.

The synthesis session

If you have scattered past context in conversations with multiple agents, don't import it. Run a 90–120 minute interview with Claude that produces one canonical state file per project. The core prompt lives in Chapter 9. Synthesis beats import every time.

Per-project rules — AGENTS.md, CLAUDE.md, or Lovable project instructions

Path A — AGENTS.md at each repo root for cross-agent portability. CLAUDE.md for Claude-Code-specific rules, hooks, subagents. Short: stack, conventions, where the state file lives, what can be touched, what cannot, receipts required.

Path B — Lovable's project instructions pane. Short: point at standing orders and state file in Basic Memory via Personal Connector; state the non-negotiables; name verification style.

In both cases, the instruction points at the canonical sources — don't duplicate content. Duplication is where drift starts.

How built-in personalization composes

Claude Desktop has three built-in layers: Profile Preferences (account-wide), Project Instructions (per-project), Styles (format/tone). These are UI settings stored in Anthropic's cloud.

Profile Preferences — shorter, more general. Language, expertise level, communication style.
Project Instructions — short pointer: "Read standing orders from Basic Memory first, then the state file." Don't duplicate.
Standing orders and AGENTS.md / Lovable instructions — portable across agents.
Styles — format and tone only.

The layered approach means you can try any new agent and it picks up the same two files and gets up to speed. Nothing is locked to Claude.

Part V

Runtime

Three chapters. What happens after setup is done and real work begins. How to verify the agent did what it claimed, how to read production, how to keep bridges honest.

Chapter 10Verify gates

Correspondence at rest is architecture. Correspondence in motion is a set of gates — specific moments where you stop, look, and confirm before continuing.

The four gates

  1. Plan gate. Before non-trivial action, the agent writes what it will do. In Claude Code, plan mode. In Lovable, the plan-first prompt. You approve or revise before any write.
  2. Execution gate. As the agent carries out the plan, raw tool output appears inline. Build logs, exit codes, migration output. Evidence that what was claimed is what happened.
  3. Verification gate. Post-execution, you confirm by looking. Dashboard, preview URL, database view, Supabase Studio. State should match what the plan said.
  4. Memory gate. End of session, propose state file updates for review. Pólya's Look Back operationalized.

No gate means no correspondence. A session without gates is an unverified claim.

The receipts clause, restated

  1. Final command output, pasted raw. Last 10–30 lines of whatever tool ran — enough to see success markers and absence of errors. Not a summary.
  2. A dashboard view or URL with a timestamp. Stripe event at 14:32. Vercel deployment abc-123. Sentry issue #4567.
  3. For UI changes, a screenshot or canvas preview. The page, after the change, loaded in a real browser.

Put this in CLAUDE.md (Path A) or Lovable project instructions (Path B). Enforce via hooks where possible. "Done" without receipts is a format error.

When a gate fails

Gate failure is always the same pattern: declared intent, execution output, and visible surface disagree. Agent said X would happen; output shows Y; dashboard shows Z. Stop. Run the Stop prompt. Investigate before acting. Most gate failures are recoverable if caught in the gate; none are if missed and compounded by the next action.

When to skip gates (rarely)

For purely read-only work — answering a question, drafting a paragraph, reading a file — gates are overhead without benefit. Anything that changes state goes through all four. Don't negotiate. A migration is stateful. A deploy is stateful. A webhook config change is stateful. Especially if you're in a hurry — hurry is when gates save you.

Chapter 11Production and deploy

Production is four surfaces you should read at a glance. Errors, behavior, uptime, deploys.

  • Errors — Sentry. What's broken right now, for whom, how often. Grouped by fingerprint.
  • Behavior — PostHog. What users actually do. Flags, replay, funnels, LLM traces.
  • Uptime — Better Stack. Is the site up? Response time? Regional outages?
  • Deploys — Vercel (Path A) or Lovable (Path B). Build logs, environment snapshot, Instant Rollback.

The five-minute morning check

  1. Sentry unresolved issues in last 24h — new ones? Spikes?
  2. PostHog — signup and active-users trend over 7 days. Anomalies?
  3. Better Stack — any monitor failures in 24h?
  4. Deploy surface — any deploy since yesterday? Post-deploy error spike in Sentry?
  5. Stripe — failed webhooks in 24h? Red indicators in Events view?

Can be scripted. An agent can do this via MCP and summarize in one paragraph. Do it before any agent work that modifies anything.

The moment of a production incident

You need to know four things, in order:

  1. What broke? Sentry fingerprint, stack trace, affected users. Raw.
  2. When did it start? Compare to last deploy. If timestamps line up, deploy is prime suspect.
  3. How many users? Sentry volume + PostHog funnel drop.
  4. Can I roll back? Instant Rollback to previous deploy is almost always correct first move if timestamps implicate the deploy. Data fixes come after.

Rolling back is not defeat. It's restoring last known good state while you investigate.

Path A · The invisible git protocol

For frontend work on Path A, GitHub becomes silent infrastructure the agent manages. Your review surface is chat, a before/after screenshot, and a preview URL. You never read a diff.

Three Vercel CLI commands replace the entire PR cycle:

vercel deploy --yes --no-wait   # preview URL
vercel promote <url> --yes      # promote to production
vercel rollback --yes           # instant alias flip

Verification discipline: the agent takes a before screenshot, makes the edit, takes an after screenshot, and opens both side by side before asking for approval. The before state is not optional — without it, correspondence cannot be checked.

Known issue (April 2026): Playwright MCP versions 0.0.56–0.0.61 break tool discovery in Claude Code 2.x. Pin with claude mcp add playwright npx @playwright/mcp@0.0.41.

Path B · Lovable's deploy cycle

Lovable owns deploy. Its build and preview happen server-side; the preview URL updates inline as you prompt. When you want production live, you promote in Lovable's UI. GitHub syncs the result as a safety net; you never open it.

Verification discipline: watch the preview as the change happens. If something is off, iterate in the same canvas before promoting. The rollback is also in Lovable — earlier versions are kept and restorable.

Chapter 12Bridges and supply chain

Your stack has more bridges than tools. Each bridge is a correspondence that must be maintained by you because no vendor owns both sides. Below it all is the supply chain, which is where attacks silently arrive.

Every bridge is a correspondence

A signup flow spans Supabase Auth, Postmark (confirmation email), the deploy target (API route), Stripe (if billing), Sentry (if errors), PostHog (if events). That's six services on one user action. Each hand-off can silently fail.

The three-trait test for bridges

A bridge is trustworthy when its visible surface shows all three:

1. Timestamps on every event — second-accurate.
2. Status colors — green or red, red visible without scrolling.
3. Expand-and-replay — click any event, see full payload, re-send.

Missing any → trust is assumed. Assumption is where bridges fail silently.

Stripe Events, Supabase Logs, Postmark Activity all pass. This is why they're Tier S/A — failure is visible by looking.

Patterns for the three common bridges

Webhooks. Signed secret (HMAC) so you know sender is real. Idempotency so double-delivery doesn't cause double-billing. Sender's Events view open during any testing. When a webhook fails, it's visible in two places: the sending service's Events view, and your app's logs (Sentry for exceptions, your own logs for caught failures). They should agree. When they don't, your handler is doing something the sending service can't see.

OAuth. Token lifecycle is the correspondence. Grant shows both in your app (user signed in) and in the provider (active session). Revoke should appear in both within minutes. Test quarterly — revoke in provider, verify app sees it gone and handles gracefully.

Scheduled jobs. The job platform has a dashboard; your app has logs. When a job runs, both should have a record. When you need to know whether last night's billing ran, you should be able to click in the job dashboard, see the status, and cross-reference the resulting rows in Supabase Studio.

Supply chain (Path A primarily)

Below every tool is a dependency graph you don't control. 2025's Shai-Hulud worm infected 700+ npm packages, 27,000+ repos, exposed ~14,000 secrets — from a single poisoned update. Defense, in order of leverage:

  1. OrbStack containerization. Every npm install or pip install of untrusted dependencies runs inside a container with explicit volume mount. Mac filesystem outside project is inaccessible to install scripts.
  2. Socket.dev. Scans dependencies before installation. Flags supply-chain risk, malicious patterns, phone-home behavior.
  3. Little Snitch in Alert mode. Every new outbound connection prompts for approval. Unexpected calls to unfamiliar domains are the signal to investigate.
  4. Git commit before install. If something misbehaves, roll back cleanly.

Path B mostly externalizes this — Lovable's servers handle dependency installation, and their security engineers are responsible for it. Your exposure is to their supply-chain posture, which is itself a bridge to verify via their security documentation and compliance attestations (SOC 2, ISO 27001). Your own Mac stays cleaner because less is installed on it.

Part VI

Resilience

One chapter. The cadences that keep correspondence from decaying, playbooks for when something breaks, and an honest map of the blindspots nobody has yet solved.

Chapter 13Cadences, playbooks, blindspots

Weekly cadence · 15 minutes

  • Backup verified restore. One file. Date logged.
  • YubiKey enrollment checklist — any new services this week? All have both keys?
  • Correspondence audit via the prompt in Chapter 9. Any drift surfaced?
  • State files — agent proposed updates from the week approved and saved.

Monthly cadence · 45 minutes

  • Dashboard screenshots. Capture dashboard-only configuration you can't script. One per critical tool. Stored in Basic Memory.
  • Recovery-code envelope review. Any new services? Any recovery codes rotated?
  • Dependency audit (Path A). Review Socket.dev findings. Patch or document.
  • Bitwarden password audit. Duplicates, weak passwords, breaches.

Quarterly · 2 hours

  • Full restore drill. From backup, into a test environment. Open the restored files. Confirm they work.
  • OAuth token revocation test. Revoke a token; verify app sees it gone.
  • Paper envelope update. Current recovery codes, service inventory, wind-down instructions.
  • Stack re-evaluation. Any tools now Tier D? Any Tier D now Tier A? Run the evaluation test in Chapter 4.

Playbooks — when correspondence breaks

The agent claims done; reality disagrees. Stop. Run the Stop prompt. Ask for raw output (Invariant 3). If the raw output contradicts the claim, the attestation was false — no punishment, just re-plan. If the raw output confirms the claim but reality still disagrees, you have a bridge failure — check the three-trait surfaces of every service in the chain.

You made a change and something else broke. Check deploy timestamp vs error start time. If they align, roll back. Data fixes come after; revert the bad deploy first to stop the bleeding.

You can't find the right note in Basic Memory. Use search_notes with the content word you remember, not the topic word. If you still can't find it, you didn't write it — the memory was from a conversation, not the vault. Write it now.

An agent session has gone sideways — rule violations, bad output, drift. Invariant 2 in action. Start a fresh session, re-load standing orders and state file, re-describe the task. Don't try to fix the drifting session; kill it.

A tool has silently deprecated an API or changed behavior. Run the correspondence audit. When the agent surface and GUI disagree, assume the GUI is current and the agent's model is stale. Update via docs; re-verify.

You lost access to a critical account. Envelope. Recovery code. Hardware key backup. This is what Day 1 and Day 2 were for.

Blindspots nobody has yet closed

Honest list of things this book doesn't solve because nobody has:

  • Cross-domain agent-action audit. No product provides "what did the agent touch today across Photos, Documents, Notes, Tasks, Code, Email." When one appears, adopt it.
  • Code-level verification for non-coders. You can verify behavior via live site + Playwright + Sentry. You cannot verify subtle semantic correctness of generated code. The gap is real; no tool closes it.
  • Research-citation hallucination. Agents cite papers that don't exist. Citation-grounding against CrossRef / Semantic Scholar / OpenAlex before insertion is the mitigation; no mainstream agent does this by default.
  • Learning comprehension verification. Retention ≠ understanding. Card-quality audit not yet a product.
  • Silent-reconciliation failure. Agent acknowledges a contradiction at session start, then reverts to default at decision time. Mitigation: re-consult state files at decision points, not just session start. Prompt: "quote the passage that supports this recommendation."
  • Biased-narrator weighting. Framework creators sell their own tools. Weight quotes from vendors as enthusiasm, not neutral observation; cite who benefits from a claim being true.
  • Multi-device story. This book is written around a primary macOS laptop. Phones, tablets, second workstations each need their own correspondence treatment not covered here.
  • Local-only air-gapped fallback. What happens when internet goes out for a week? Offline-first workflows are not this book's current scope.

CCase study — one operator's actual stack

The book's author is a solo SaaS founder in Witzenhausen, Germany. Non-coder. Two active projects: the SaaS product, and this book.

What works. Supabase as foundation. P/K/e ranking algorithm running correctly, verified with hand-calc receipts against synthetic data. RLS policies enforcing access control. Basic Memory Cloud holding vision, decisions, reversed paths, session notes. Claude Desktop for thinking; live MCP for measurement. YubiKey on every critical account. 3-2-1-1-0 backup passing monthly restore.

What didn't work at first. The sign-up page broke silently on the Next.js frontend — the submit handler fired no network traffic, a failure pattern that only Playwright-style observation caught. Multi-week paralysis trying to debug it with Claude Code against a GitHub repo the author couldn't read. The correspondence concern was real: GitHub was a fully opaque surface in a stack that required trusting it.

The reversal. Moved to Path B (Lovable + Supabase). The sign-up bug disappeared because the broken frontend was rebuilt on a visible-canvas surface. Code still exists on GitHub via Lovable's sync; author never reads it. Stack collapsed from five tools (Next.js + Vercel + GitHub + Claude Code + Supabase) to two (Lovable + Supabase) plus Basic Memory + Claude Desktop as shared context layer. The seven-day setup condensed to four days.

What this reveals about the book. Earlier versions classified AI-native builders as Tier D uniformly. The evidence showed the category had split: Lovable in April 2026 has portable output (bidirectional GitHub sync), native Supabase integration (backend is real, not proprietary), MCP personal connectors (your memory travels), and enterprise compliance (SOC 2, ISO 27001, GDPR). That's not Tier D. This book's v4.0 reclassification reflects real operator evidence, not vendor marketing.

What's still open. Verifying the full pairwise-comparison interaction (binary matchups, tier grouping, diamond↔fire slider) on Lovable's canvas at equal quality to what a hand-rolled Next.js build would produce. Custom interactive logic is where AI-native builders struggle most. If Lovable fails here, the fallback is Builder.io Fusion (visual canvas on top of Next.js) or return to Path A with a hired collaborator who reads code.

The point of including this: no book written by a single operator can claim to be a view from above. This one is a view from inside, corrected as evidence arrives. Read it that way.

DGlossary

Correspondence
The property that a capability's visible surface and its agent interface point at the same state.
Underlap
A capability that exists only on one side — visible but not agent-driveable, or agent-driveable but not visible.
Overlap with drift
Both surfaces exist but have fallen out of sync. The more dangerous failure because it produces false visibility.
Bridge
A connection between two tools that no single vendor owns both sides of. You maintain the correspondence across it.
Three-trait test
Timestamps + status colors + expand-and-replay. A bridge's visible surface passes the test or it's not trustworthy.
Tier S / A / B / C / D
The ranking framework in Chapter 4. S is near-perfect correspondence; D is one-sided and usually dangerous.
Path A / Path B
Path A is Next.js + Vercel + Claude Code + GitHub. Path B is Lovable + Supabase. Same principle, different composition, different setup length.
MCP
Model Context Protocol. The standardized interface between AI agents and external tools.
Pólya's four phases
Understand, Devise, Carry out, Look back. The shape of every problem-solving session.
Verify gate
A specific moment — plan, execution, verification, memory — where you stop, look, confirm before continuing.
Receipts clause
"Done" requires raw output, a timestamped dashboard view, and for UI changes a screenshot. No receipts, no completion.
Mutability split
Immutable intent (decisions, rationale, history) belongs in Basic Memory. Mutable facts (schema, deploys, bugs) belong in live MCP queries, not cached in the vault.
