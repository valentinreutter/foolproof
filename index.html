The Manual · v3.0

Foolproof

The correspondence principle for working with an AI agent — a setup and operating manual for solo work in 2026, for the person who cannot read code.

The principle in one sentence:
The visual surface you can verify and the agent interface that controls your work must be two views of the same state — every capability reachable from both sides, nothing living in only one layer, nothing split across two layers that drift.
V3.0 · April 2026 · A living document

This book is how a person who cannot read code builds, operates, and recovers a real working system on that principle. It names what correspondence guarantees and what it doesn't, recommends a specific stack as of April 2026, teaches Pólya's four phases as the shape of every session with an agent, and maps the blindspots no vendor has yet closed.

Straight through once for the thesis and the shape. Then as reference: the tier ranking in Chapter 5, the prompt library in Chapter 11, and the playbooks in Chapter 17 are the pages you return to.

Parts I and II are theory. They take an hour. Part I is the principle of correspondence and its structural failures. Part II ranks the tools by how well they preserve correspondence in 2026.

Part III is the build. It takes a week, about two hours a day. You follow the seven-day setup. At the end, your machine is a safe substrate for agent work.

Part IV is the craft. It is the deepest part of the book. How to think about prompts and context the way Pólya thought about problems. How to get the most out of Claude Desktop and Claude Code. How to use Basic Memory as a semantic knowledge graph rather than a folder of notes. What to put in AGENTS.md and how to compose it with Claude's built-in personalization. The prompts you will use every day, organized by Pólya's four phases.

Parts V and VI are daily operation. How correspondence is preserved at runtime, across integrations, and through deploys. The eight domains where solo work happens.

Part VII is resilience. The weekly and monthly cadences that keep correspondence from decaying. The playbooks for when something goes wrong. An honest map of blindspots nobody has yet solved.

The appendix is reference. Substitution tables for every named tool. A jurisdiction-specific section on operating in Germany. Glossary. Further reading. Colophon.

This book names specific services — Porkbun, Claude, Supabase, Basic Memory — as worked examples with a specific person in mind: a solo SaaS founder in Germany, April 2026. The structure is universal. The principles are evergreen. The tools are timestamped and marked as such. When the tools change, you substitute by the tier framework in Chapter 5; the principles don't move.

A note on who this book is for

This book assumes you cannot read code fluently. If you can, you will find the early chapters slower than necessary — skim to Chapter 5. The book never hides depth from a non-coder; it also does not slow down for a technical reader in the later chapters. The arc runs from accessible to dense.

Part I The principle

Four chapters. Why correspondence is the right idea, what its two failure modes are, what it does not claim to fix, and where it doesn't apply at all.

Chapter 1Correspondence

You cannot read the code your agent writes. Not with fluency. That's the constraint everything else in this book works around.

The naive answer is: trust the agent. Trust the model. Trust the tests if they pass. Trust the deploy. This answer breaks the first time something silent goes wrong — the migration that ran but skipped a column, the webhook that fired but never arrived, the dependency that installed and quietly phoned home. Trust scales badly when you cannot read the thing you're trusting.

The working answer is correspondence.

The principle

Every capability in your system must have a visible surface a human can look at and verify, and an agent interface an AI can read and act on. Both must point at the same underlying state. No capability lives in only one layer. No capability is split across two layers that drift.

"Visible surface" means a GUI, dashboard, admin panel, or file tree a non-coder can open and understand. "Agent interface" means an API, CLI, or MCP server the agent can call. "Same underlying state" means both are projections of one source of truth, and when one changes, the other sees the change.

Two surfaces. One state. That is the whole idea. The rest of the book is how to apply it in practice.

You look at the visible surface; the agent reads the agent interface; both are views of one underlying state.

The two failure modes

Underlap is a capability that exists in only one layer.

A bare VPS is the canonical underlap. The agent has full SSH access — it can read any file, install any package, start any service. There is no corresponding visible surface. If something breaks overnight, you have no way to look at the machine; you can only ask the agent what happened and have to believe the answer. Every decision point has a single, un-cross-checked source.

A no-code builder like Webflow is the opposite underlap. The visible surface is rich — you can drag, drop, see. The agent interface is thin or absent. You can inspect but not audit programmatically; you cannot script the system's own state without leaving the platform.

Both are half-systems. Both force you to operate blind on the missing side.

Overlap is a capability that exists in two layers that drift apart.

Two Stripe products with the same name but different prices. A value shown in the Vercel dashboard that doesn't match what the deployed function reads from process.env. A webhook URL configured in one service that points at an endpoint the other service moved six months ago. Each of these is a correspondence that broke. The system has two views but they no longer agree.

Overlap is more dangerous than underlap because it creates the illusion of visibility. You look, you see something, you assume it matches reality. It doesn't.

Before and after — what correspondence buys you

Before. An operator runs their SaaS on a €20/month VPS. The agent has root. The database is Postgres accessed via psql. Backups are cron jobs. The operator checks uptime by tailing logs over SSH. One morning a user reports charges that didn't go through. The operator asks the agent to investigate. The agent reports: "the payment webhook handler crashed at 03:14 UTC; I've restarted the service." The operator has no independent view. No dashboard. No events log. No replay capability. The agent's report is the only record, and the failed charges are unrecoverable without a backfill the operator cannot independently verify.

After. Same operator, correspondence-preserving stack. Payments run through Stripe (dashboard + API + MCP, all hitting the same event stream). Database is Supabase (Studio GUI + SQL + MCP, all hitting the same project). Logs go to Sentry (dashboard + MCP). When the same failure happens, the operator sees it in Stripe's Events view at 03:15, clicks the failed webhook, expands the payload, hits "replay" — all without reading code. The agent can do the same replay via MCP while the operator watches. The two layers agree on what happened; recovery is a click or a command, and both surfaces confirm it succeeded.

The difference is not that the second setup is safer in the abstract. The difference is that failure is visible to the operator, and recovery is actionable from either side.

Where the principle works and where it doesn't

Correspondence applies cleanly to resources — the things your system holds. A database row, a user account, an environment variable, a Stripe product, a DNS record, a deployed build. For every resource in your system, you can have a visible surface (dashboard, Studio, admin) and an agent interface (MCP, API, CLI) pointing at the same state. Change one, the other sees it. This is where the principle delivers fully on its promise.

Correspondence does not apply cleanly to behavior — the things your system does. Your Next.js .jsx files and the rendered page a user sees are not two views of one state. They're connected by a build pipeline (Vercel compiles the code, a browser runs it). You cannot cross-check the code against the rendered page because you cannot read the code. What the agent writes in the .jsx file and what you see when you open the preview URL are related — but they're not corresponding surfaces of the same state.

For behavior — which for a non-coder means "the frontend" and "custom business logic" — the best available isn't correspondence. It's verification by observation: you click the preview URL, try the feature, look at the Playwright screenshot, trust a subagent reviewer. These are compensations for the missing correspondence, not instances of it.

Where this matters: the recommended stack is strongest when most of the system's work lives in resources — user accounts, data, payments, emails, deploys, observability. A backend-heavy SaaS with a thin frontend gets almost full correspondence coverage. A frontend-heavy product with complex custom UI gets weaker coverage, and the non-coder operator path is genuinely harder there. Know which side of the line your product sits on.

What correspondence is not

Correspondence is not closure. A correspondence-preserving stack can still be compromised, still lose data, still fail silently at a network edge. The principle doesn't guarantee outcomes; it guarantees that when something goes wrong, you can see it, and when you act on what you see, the action is real.

It is not the same as "having a dashboard." Many tools have dashboards that don't correspond to anything the agent can touch.

It is not the same as "having an API." A headless service with a complete API and no GUI is agent-only; the operator cannot verify without writing code.

It is the conjunction. Both layers, the same state, across setup and runtime.

Why this book exists

Most writing on AI-agent safety is about containing the agent. Sandbox it. Limit its tools. Review every action. That advice is good and this book uses it. But it treats the agent as the problem and the human as the verifier, and it assumes the human can read what's being verified.

This book starts from a different premise. The human cannot read what's being verified. They need a system where verification is a look, not a read. Correspondence is the structural property that makes verification-by-looking possible.

The rest of the book is: how to rank tools by how much correspondence they give you, how to compose tools so correspondence survives their integration, how to build the whole system in a way that itself preserves correspondence, and how to operate under correspondence once it's running. Plus an honest map of where correspondence cannot be preserved today, so you know where to compensate.

Correspondence is not the only way to work with AI agents. It is the only way I have found that a non-coder can trust.

Chapter 2Overlap, underlap, composition

Correspondence is a property of a single tool. Composition is what happens when you chain tools together. Composition is where most real systems fail.

The two failures named precisely

Underlap on the agent side. The capability is visible but not agent-driveable. A Notion database the official MCP can read but not fully mutate. A Webflow site the operator can edit by dragging but the agent cannot audit programmatically. A Fastmail inbox with a rich GUI and no agent-inspectable audit surface. The operator can see; the agent is blind.

Underlap on the visible side. The capability is agent-driveable but not visible. A bare VPS with SSH only. A headless CMS with a complete API and no admin UI. Most CLI-first infrastructure. The agent can act; the operator cannot verify without writing code.

Overlap with drift. Both layers exist, but they fail to stay synchronized. The most common patterns:

  • Dashboard value and API value diverge. A Stripe price changes in code; the dashboard label was never updated.
  • .env.local on the laptop, .env on a staging server, and Vercel's environment variables UI each hold a different value for the same key.
  • The webhook URL configured in a third-party SaaS points at an endpoint the main app moved months ago.
  • A schema migration ran in production but not in preview; the two environments now disagree about what columns exist.

Overlap is worse than underlap because it produces false visibility. The operator opens the dashboard, sees a value, acts on it — but the real state is different. The loop between looking and acting is broken without any error appearing.

Every boundary is a correspondence

A solo operator is not running one tool. They are running a dozen or more, wired together: domain → DNS → edge function → application server → database → background jobs → email → analytics → error tracking → billing. Each tool may individually have tight correspondence. The composition rarely does.

Each arrow above is a bridge. Each bridge is a place where correspondence must be maintained by you, because no vendor owns both sides. The webhook from Stripe to your app. The OAuth handshake from your app to a user's Google account. The sync between your database and your analytics. The email forwarder from your domain to your inbox. These are the places correspondence silently decays.

Chapter 14 is entirely about this. For now: the number of bridges in your system is the honest measure of how much correspondence work you are taking on.

The minimum-correspondence principle

Principle

The number of correspondences you must maintain scales with the number of tool boundaries. Therefore: prefer fewer tools with tight internal correspondence over many tools each with partial correspondence.

Concretely: a Laravel application with Filament for admin, Postgres, a Queue, and deployed as one binary to one host is one correspondence — the application's own state, visible in Filament, driveable via Artisan and MCP. A Next.js front-end on Vercel with Supabase for data, Inngest for jobs, Resend for email, Stripe for billing, PostHog for analytics, and Sentry for errors is seven correspondences, each with a bridge.

The first system is harder to build and harder to extend. The second is easier to build and harder to verify. The tradeoff is real; the recommended stack in Chapter 5 chooses the second path because in 2026 the bridges are tractable if every tool in the composition is Tier S or A. But the principle stands: if you can collapse two tools into one without losing correspondence, do it.

When overlap is acceptable

Some overlap is unavoidable and fine. A local development database and a production database are intentionally separate — they are not supposed to share state. A staging environment and production are meant to diverge. What matters is that the relationship is named and tested: you know which direction state flows, and you have a scheduled check that confirms they agree on what they're supposed to agree on.

The enforcement rule: every intentional overlap has a named direction of truth (e.g., "production is canonical; staging pulls from production nightly") and a weekly cadence check (Chapter 16) that compares the two.

Checklist for a new tool

Before adding a tool to your stack, answer:

  • Does the tool have a GUI a non-coder can open and use?
  • Does it have an agent interface (MCP, API, CLI) that reaches the same state?
  • Do changes made in one surface appear in the other within a known time window?
  • If it doesn't own its own data (it's a bridge to another service), does it pass the three-trait test in Chapter 14: timestamps, status colors, expand-and-replay?
  • What does adding it take your total correspondence count to?

If the answers are all clean and the correspondence-count increase is justified by capability gain, add it. Otherwise substitute or do without.

Chapter 3Non-guarantees

A book that promises foolproofness is lying. This one does not. Before you apply any recommendation that follows, know what correspondence does not give you.

What correspondence does not do

It does not prevent compromise. A phishing attack that tricks you into authorizing a malicious MCP server bypasses every tier ranking in this book. A supply-chain worm that lands in a dependency before Socket.dev flags it will run on your machine regardless of container boundaries. Hardware-key enrollment does not help if an attacker gets physical access to an unlocked laptop.

It does not eliminate silent bridge failure. Even Tier S tools have bridges to Tier C systems somewhere. Your Stripe webhooks land at a Vercel edge function that calls a Supabase function that writes to a third-party API with no audit surface. Somewhere in that chain, silence happens. The three-trait test catches most cases; it does not catch all cases.

It does not self-verify agent output. Code an agent writes can pass every test, ship clean through preview, and be wrong in production. Content an agent writes can be grammatically flawless and factually false. Correspondence makes the agent's actions visible; it does not make the agent's judgment correct.

It does not extend to frontend code for a non-coder. The principle applies to resources (database state, accounts, deploys, secrets), not to behavior produced by code you cannot read. For frontend work, the visible surface (rendered page) and the agent interface (.jsx files) are connected by a build pipeline, not by two views of one state. What you have there is verification-by-observation (preview URL, Playwright screenshot, subagent review), which is weaker than correspondence. Name the gap; don't pretend it isn't one.

It does not remove human judgment. Every verify gate in this book (Chapter 12) requires you to look at something and decide. Correspondence reduces how much you must decide. It does not replace the deciding.

It does not cover devices beyond your primary laptop unless you add them. This book is written around a primary macOS laptop. A phone, a tablet, or a second workstation each need their own correspondence treatment. That work is not in the seven-day setup.

It does not replace legal, financial, medical, or tax advice. If an agent helps you file taxes, the tax liability is still yours and your tax advisor's. Appendix B on Germany-specific compliance (GoBD) is context for conversations with professionals, not a substitute for them.

It does not make correspondence-rated tools infallible. Supabase can have an outage. Vercel can fat-finger a config change. Cloudflare can drop traffic. The tier ranking is about how detectable and recoverable failures are — not about whether failures happen.

What correspondence does do

It makes failures detectable by a non-coder at near-zero latency, using the surfaces you already have open.

It makes recovery actionable from either the visual or the agent side, since both reach the same state.

It makes decisions traceable — the same event appears in the logs you read and the logs the agent reads, with the same timestamp and payload.

It reduces the footprint of "trust me" in the system. Not to zero. To a known, bounded, auditable surface area.

The honest residual-risk ledger

The rest of the book recommends specific tools and patterns. For each, here is what residual risk remains even after the recommendation is applied correctly.

Identity. A phishing attack that intercepts your hardware-key prompt can still authenticate as you. YubiKeys are phishing-resistant, not phishing-proof.
Execution. A malicious dependency installed before Socket.dev flags it runs with your user privileges inside the container. Little Snitch catches outbound; it does not catch local damage.
Data. Supabase's point-in-time recovery restores the database, not the application logic that corrupted it. Bad writes are reversible; the bad deploy that caused them is not automatically.
Deploy. Vercel Instant Rollback fixes bad code; it does not fix the bad data the bad code wrote. Backups handle data; Git handles code; nothing handles the gap between them automatically.
Bridges. Webhooks can pass the three-trait test and still be slow enough to cause user-visible latency. Correspondence is about visibility, not timeliness.
Supply chain. Socket.dev, Little Snitch, and OrbStack compose well but have never been tested against a zero-day dependency attack on your specific machine at your specific time.
Backups. Three-two-one-one-zero with Object Lock and a verified weekly restore covers known failure modes. It does not cover a sophisticated attacker who compromises your account before you notice and has time to corrupt restore sets.
Agent. The agent can hallucinate, drift from instructions mid-session, or silently reconcile to your preferences in ways you didn't sanction. The prompts in Chapter 11 reduce this; they do not eliminate it.
The book itself. The April 2026 tier ranking reflects tools as of writing. MCP servers ship, deprecate, and change scope constantly. Re-verify before major decisions.

Read the rest accordingly

Everything that follows is written assuming you have read this chapter. When the book recommends a Tier S tool, it means: this is the best available in 2026, named failures included. When it says "verify weekly," it means: the failure mode exists, the check catches the usual cases, you are on your own for the unusual ones.

Correspondence is the best structural property I know of for an operator who cannot read code. It is not magic. Act accordingly.

Chapter 4When not to use AI

Correspondence is a discipline for problem-shaped work. Not all meaningful work is problem-shaped. This chapter names where the principle doesn't apply — and therefore, where the agent doesn't belong.

Pólya's four-phase method, which is the spine of Part IV, was written for mathematical problems: questions with discoverable answers. Its power comes from forcing you to name the unknown, identify the data, specify the condition, and then search for the connection. This works beautifully for software, research, decisions with clear tradeoffs, learning specific skills, and administrative processes — anything with an identifiable unknown.

It fails, sometimes spectacularly, for work where there is no unknown to find.

Where correspondence does not apply

Emotional processing. Grief, anger, disappointment, joy, doubt. These are not problems. They do not have solutions. The work is to feel them, to let them change you, to move through them. An agent that tries to help by reframing them into problem language — "What's the underlying issue? What would resolution look like?" — does harm. It trains you to bypass the feeling instead of meeting it.

Creative generation for its own sake. A poem written to discover something you didn't know you thought. A song. A sketch. A piece of writing that exists because the process of making it is what you wanted. The output isn't the point; the making is. An agent that offers to write this for you eliminates the purpose.

Exploratory conversation with another human. The meandering chat with a friend that goes nowhere important and means everything. The coffee with a mentor where you don't know what you want to talk about until an hour in. These are not inefficient meetings waiting to be optimized; they are the texture of thinking together.

Embodied attention. Cooking a specific meal without a recipe because you're paying attention to the ingredients. Walking somewhere new without a map. Sitting with a difficult book and not looking anything up. These are practices in paying attention; the friction is the point. Replacing the friction with agent shortcuts is not help.

Personal journaling. Writing to yourself, for yourself, about yourself. A journal that is co-authored by an agent isn't a journal anymore — it's a transcript of a conversation about you. The reflective privacy is the function.

Relationships. The cadence of checking in on someone you love. The apology that lands because you thought about it. The difficult conversation that works because you went into it uncertain. Agent-drafted messages to intimates are perceptible as such, and perceiving them damages the relationship more than an awkward human message would.

The diagnostic

When in doubt, ask: is there an unknown here? A definite thing I'm trying to produce, learn, decide, or verify? If yes, the work is problem-shaped and the agent fits. If no — if the point is the process, the feeling, the attention, the relationship, the making — the agent does not belong.

Where correspondence partially applies

Some work is partly problem-shaped. Writing an essay has an unknown (the finished essay) and a process (writing, which is often where the real thinking happens). Learning a new domain has a goal (competence) and a practice (the time spent struggling, which is how the competence forms). In these cases, use the agent for the problem parts — research, citation checking, structure, editing — and protect the practice parts from being optimized away.

A useful rule: if removing the agent from a step would damage what you're actually trying to do, remove the agent from that step. If removing it would only cost you time, the agent is fine.

Why this chapter is early

I put this chapter in Part I, next to the principle itself, because the book that follows teaches you to use an agent well across many domains of work. You will get better at it. The risk of getting better at it is that you start using the agent for things where it doesn't belong, and the tool that was help becomes the tool that hollows you out. The principle of correspondence is the principle of problem-shaped work. When you notice work that isn't problem-shaped, you notice that the principle does not apply, and you put the agent down.

The agent is a tool for questions with answers. Much of life is not that. Keep the two separate.

Chapter 5Tiers and composition

This chapter is the book's main reference. If you read only one chapter in full, read this one. It ranks every category a solo SaaS operator needs, recommends a specific composition, and names the honest alternatives.

Timestamped stamp

This ranking reflects the state of tools as of April 2026, after the MCP Apps extension shipped (January 2026) and Anthropic donated MCP to the Agentic AI Foundation (December 2025). The tiers are an evergreen framework. The assignments of tools to tiers are a snapshot. Re-verify before any major decision; run the evaluation test at the end of this chapter whenever a new tool appears.

The tiers

Tier S — near-perfect correspondence. Remote MCP server with OAuth, same state as the dashboard, agent can do ~95%+ of what the GUI can do, changes in either surface appear in the other within seconds. The bar: if you close the dashboard and do everything through MCP, then open the dashboard an hour later, you see exactly what you would expect. And vice versa.
Tier A — strong correspondence, named gaps. Remote MCP or complete API, most operations parity, a small set of GUI-only features (usually org settings, destructive actions, or inherently visual tasks like watching a session replay). Gaps are documented and known.
Tier B — partial, bridge-dependent. Has an API the agent can reach but not a first-party MCP server, or the MCP covers only a slice. Composing this tool into the stack requires writing your own bridge and maintaining its correspondence yourself.
Tier C — broken divergence. Has both a GUI and an API, but they have diverged enough that the agent's view and the human's view disagree in operation. Notion, Jira, most old-guard SaaS platforms. You can bolt on community MCP servers, but they drift from vendor changes within weeks.
Tier D — one-sided, usually dangerous. Either GUI-only with no meaningful agent interface (most classic no-code), or agent-first with opaque state (Bolt.new's build process, Lovable's build process, Replit Agent). The "AI-native builders" that generate deploys you cannot inspect in a familiar dashboard are the most dangerous new entrant in 2026 because they feel productive while eliminating correspondence.

Integrated-platform path. Fewer tools, fewer bridges, single correspondence per domain. Laravel with Filament for admin and Postgres on one host. Phoenix LiveView with Postgres. Reduces vendor exposure, reduces correspondence (most self-hosted tools lack rich MCP), increases operational burden. Good if you value data sovereignty above verification ease. Not the default here.

How to evaluate a new tool

When a new tool appears — and they do, weekly — apply this test:

  1. Does it have a visible surface a non-coder can open?
  2. Does it have an agent interface that reaches the same state? MCP server? API with a published OpenAPI spec? CLI?
  3. Do the surfaces agree? Make a change in one; verify it appears in the other within 60 seconds.
  4. Does the agent interface have authentication that survives plan mode? OAuth remote MCP beats local MCP beats scripted API beats scraped UI.
  5. What tier do the bridges to your existing stack land on?
  6. What is your total correspondence count after adoption, and is the capability gain worth the increase?

A tool passing the test can be adopted. A tool failing any step should be substituted, waited on, or accepted as a Tier B-or-worse dependency with documented gaps. When in doubt, default to not adopting. Each tool added is correspondence work; each correspondence can decay; decayed correspondence is the silent failure mode this book is structured to prevent.

Two features that sound alike but aren't

Registrars offer both URL forwarding (redirects web visitors from one domain to another) and email forwarding (relays mail to an inbox you already have). These are different features. You want email forwarding — or, cleaner for 2026, Cloudflare Email Routing. Paid email hosting at the registrar is a third option and not needed for this setup.

1.1 · Buy the domain at Porkbun (10 min)

  1. Go to porkbun.com. Create an account with 2FA enabled.
  2. Search for your domain. yourname.com or yourname.xyz works fine — this is infrastructure, not branding.
  3. Buy it. Roughly €10/year. Enable auto-renew immediately; a lapsed domain is the worst identity emergency possible.

1.2 · Decide your DNS path (15 min)

You have two reasonable choices. Both work. Best practice in 2026 is path B.

Path A — keep DNS at Porkbun, use Porkbun email forwarding. Simpler. Fewer accounts. Porkbun's email forwarding is free for up to 20 aliases, rate-limited at about 50 messages per hour per alias. Good for a lean starting state.
Path B — move DNS to Cloudflare, use Cloudflare Email Routing. Best practice. Cloudflare's DNS is faster and more inspectable. Email Routing is free, supports wildcards (everything@yourdomain.com to one inbox), and has better deliverability analytics. You point Porkbun's nameservers at Cloudflare's; Cloudflare manages everything downstream.

Path B requires a Cloudflare account (free) and one DNS propagation wait (a few hours). If you want the simplest working setup, take path A and migrate later.

1.3 · Create your main forward (10 min)

Whichever path you took, create the same logical forward. Use hello@ — generic, professional, privacy-friendly. Not your legal first name.

Path A (Porkbun): Domain Management → [your domain] → Details → Email Forwarding. Box fields: When someone sends email to hello, deliver it to your existing email. Click Create.

Path B (Cloudflare): Cloudflare dashboard → Email → Email Routing → Routes → Create address. Field: hello@yourdomain.com → forward to your existing email. Click Save.

Chapter 7Pólya and the discipline of understanding

The single deepest skill in working with an AI agent is not prompt craft. It is the discipline of understanding what you are actually trying to do before asking for help. George Pólya described this discipline in 1945. Nothing has changed since.

Pólya's four phases

In How to Solve It, the Hungarian mathematician Pólya laid out a four-phase structure for solving any problem. It was written for mathematics students, and it is often taught as a mathematics method. Its power is actually far broader: every piece of problem-shaped work — software, research, a hard decision, a careful explanation — moves through these same four phases whether the person doing it knows Pólya's name or not.

  1. Understand the problem. Name the unknown (what are we producing?). Identify the data (what's given?). Specify the condition (what must be true?). Is the condition sufficient to determine the unknown? Is it insufficient? Redundant? Contradictory?
  2. Devise a plan. Have you seen the same problem in another form? Do you know a related problem? Can you restate it? Can you solve a simpler version first?
  3. Carry out the plan. Execute each step. Check each step. Can you see clearly that the step is correct?
  4. Look back. Can you check the result? Derive it differently? Use the method for another problem?

John Conway, in the foreword to the 2004 edition, names the deepest point: often the hardest part of researching a problem is understanding precisely what that problem says. Experienced practitioners spend more time on Phase 1 than beginners think is reasonable. Beginners skip to Phase 3 and solve the wrong problem.

Why this is central to working with agents

The number one failure mode in agent-assisted work is the agent solving the wrong problem. You gave a vague instruction; the agent filled in the blanks with plausible guesses; it produced beautiful code that addresses a problem you don't have. You thought the agent would clarify. It didn't. It proceeded.

The cure is Pólya Phase 1, applied to your own prompts. Before you tell an agent anything, work out the unknown, the data, and the condition.

Pólya's three Understand questions, adapted for agents

What is the unknown? What specifically am I asking the agent to produce? A working feature? A correct answer? A decision between options? Name it in one sentence.

What are the data? What is already in place that the agent needs to know about? The stack? The current state? The constraints from the last conversation? Prior decisions? What should be loaded into context?

What is the condition? What must be true of the output for it to count as success? What must not be true? What are the non-negotiables (scoped access, no secrets in logs, no destructive commands)? What are the preferences (communication style, format, language)?

Pólya then asks: is the condition sufficient to determine the unknown? If not — if there are multiple acceptable outputs all satisfying your condition — the agent will pick one, and you will be surprised at which. Refine the condition until it points at the output you want.

When you find yourself iterating with an agent for many turns on what feels like a simple task, look back. You almost certainly skipped Phase 1. Stop the session, write down the unknown, data, and condition, and start over with the answers embedded in your first message. The work is usually done in one turn after that.

Why meta-conversation counts as Phase 1

You already know this from experience: when you try to explain to an agent what you want and find yourself being asked clarifying questions back, the clarifying questions are Pólya Phase 1 happening. The agent is trying to pin down unknown, data, and condition. Good clarifying questions from the agent are a signal of a good session; they shorten the total turn count even though they delay the first code or output. Bad sessions skip this: the agent doesn't ask, and by the time you realize it solved the wrong problem, you've paid for five turns of wasted work.

If you notice an agent starting to generate output without having asked the clarifying questions, stop. Paste: "Before answering, tell me what you think the unknown, the data, and the condition are. I want to check your understanding of the problem." This is Pólya applied live.

Phase 2 — Devise — applied to agents

The agent is the one proposing plans; your job is to read them. The concrete practice is plan mode in Claude Code (Chapter 9), the plan-first prompt (Chapter 11), and the change-proposal prompt. Each is a way of forcing Phase 2 to happen visibly before Phase 3 begins.

Pólya's Phase 2 questions map almost directly:

  • Have you seen a related problem? → "What precedent or existing pattern does this resemble? Search the codebase before writing anything new."
  • Can you restate the problem? → "Rephrase what I asked in your own words before proposing a solution."
  • Can you solve a simpler version first? → "What's the smallest version of this change that demonstrates the approach? Start there."

Phase 3 — Carry out — applied to agents

"Check each step. Can you see clearly that the step is correct? Can you prove that it is correct?"

Phase 3 is verify gates (Chapter 12). The receipts clause (Chapter 11). Raw output over summary. The agent carries out the plan; you check each step not by reading the code but by looking at the evidence the agent produces — the build log, the test output, the Playwright screenshot, the dashboard state.

The principle behind every verify gate: the evidence must exist before the claim of completion is accepted. "Done" without evidence is a format error.

Phase 4 — Look back — applied to agents

This is the most-skipped phase, and it is where compounding happens. After every substantive session, ask three questions:

  1. Can I check the result? What is the specific, observable evidence that this worked?
  2. Could I have done it differently? Was there a simpler path I didn't see?
  3. Can I use the method elsewhere? Is there a pattern in how we worked today that applies to the next problem?

The output of Phase 4 goes back into your memory vault as updates to the project's state file. A decision you didn't have yesterday becomes an observation in Basic Memory today. An approach you tried and reversed becomes a [reversed] entry that future sessions will see and avoid. The end-of-session prompt (Chapter 11, Prompt 3) is the operational form of Look Back.

Skip Phase 4 repeatedly and each session is isolated. Do it even briefly and each session makes the next one better. Over a month, the difference is enormous.

Chapter 8Prompt and context engineering

Prompts date. Models change. The invariants that make prompts work don't. This chapter names the durable principles of prompt and context engineering — the ones that held in 2023, hold in 2026, and will hold in 2029 — then shows how to apply them with the tools available now.

The durable invariants

Four principles that outlast any model generation. Everything tactical derives from these.

Invariant 1: Verification must cost less than the action it verifies

If checking whether the agent did X right takes more time than doing X yourself, you will stop checking. The whole system collapses. Every workflow, prompt pattern, and tool choice in this book exists to drive verification cost down — receipts that are cheaper to skim than to re-derive, screenshots that are cheaper to look at than to re-produce, state files that are cheaper to re-read than to rebuild. When verification becomes expensive, the system is failing. Redesign until it's cheap again.

Invariant 2: Agents drift proportional to session length

Every instruction you set at the start of a session erodes as the session grows. By turn 15 or 20, the agent has begun treating early constraints as background noise. This is not a bug to be eliminated; it's a structural property of how models use context. The operational consequence: for long tasks, start fresh sessions often, each with the relevant state file reloaded, rather than one mega-session. If you notice an agent violating a rule you set at turn 3, don't remind — reload.

Invariant 3: Raw output is verification; summaries are attestations

When the agent says "tests passed," that is an attestation you must trust. When the agent pastes the last 20 lines of npm test output including the "12 passed, 0 failed" line at the bottom, that is verification. The difference matters more than you think. Agents will attest to work they didn't do. They cannot paste raw output they didn't get. Always ask for raw output when the stakes are real.

Invariant 4: Context is a budget, not free

What the agent "knows" in a session is exactly what's in its context window — nothing more, nothing less. Loading a 50-page document costs tokens that can't be spent on reasoning. Loading irrelevant context is worse than loading none, because it distracts. The skill of context engineering is deciding what to load at session start, what to load on demand, what to never load, and what to synthesize from raw material into smaller canonical files. Your state files (Chapter 10) are the explicit act of spending context budget well.

Context engineering, concretely

Prompt engineering asks: what should I say to the agent? Context engineering asks the deeper question: what should the agent already know before I say anything? Both matter; context engineering matters more because it shapes what prompts are capable of doing.

The four sources of context, ranked by how often you should spend them:

Always loaded (cheap, high-signal). Your cross-project standing-orders note in Basic Memory: who you are, non-negotiables, the tools you use. For code sessions in a specific repo, also the repo's AGENTS.md (or CLAUDE.md). Typically under 1000 words each. Always read at session start.
Loaded per project (medium, high-signal). Your project's state file in Basic Memory — current architecture, decisions made, decisions reversed, rejected approaches, open questions. Typically 500–800 words. Loaded when you start a session about that project.
Loaded on demand (expensive, high-signal). Specific documentation, past conversation retrieval, files you point the agent at. Loaded for the current task only. Not kept across sessions unless they become part of state.
Never loaded (noise). Raw transcripts of old conversations. Logs of work completed months ago. Anything that has already been synthesized into a state file — you want the synthesis, not the raw material.

The single most important context-engineering habit is synthesis over import: producing small canonical state files from your scattered past thinking rather than dumping old conversations into the agent. Chapter 10 covers the synthesis session in detail. For now: if you find yourself pasting long documents into the agent repeatedly, your state files are wrong. Fix the state files; stop pasting.

Prompt engineering, derived from the invariants

The prompts that earn their place all derive from the four invariants:

  • Pólya's Understand questions (unknown, data, condition) derive from Invariant 1 — cheap verification only works if the agent is working on the right problem. Get Phase 1 right and Phases 3–4 become easy.
  • Plan-first and change-proposal prompts derive from Invariant 1 as well — reviewing a plan is cheaper than reviewing finished code.
  • The quote-state-passage prompt derives from Invariant 2 — it counters silent drift by forcing the agent to produce evidence of the constraint it claims to be following.
  • The receipts clause and raw-output prompts derive from Invariant 3.
  • The synthesis prompt and end-of-session state update derive from Invariant 4 — they produce the compact, high-signal artifacts that future sessions load.

Chapter 11 is the prompt library. Every prompt there traces back to one of these four invariants. If a new prompt idea doesn't, it probably isn't worth adopting.

4. Hooks — enforcement as code

Claude Code supports hooks at PreToolUse, PostToolUse, and Stop. These are shell commands that run automatically and can block the agent by exiting non-zero. Four hook patterns worth installing on any serious project:

  • Stop-test-gate: before the agent claims completion, run npm test. Must pass.
  • Post-Edit reviewer: spawn the qa-reviewer subagent automatically after significant file changes.

Hooks are pure enforcement. The agent doesn't have to remember to run the formatter; the hook ensures it happens. This converts discipline into infrastructure.

Chapter 10Basic Memory as semantic knowledge graph

Basic Memory is not a folder of markdown files. It is a semantic knowledge graph — nodes with categorized observations, edges with typed relations, and agents that can traverse the graph rather than just keyword-search it. Using it like a note-folder wastes most of its value.

The primitives

A Basic Memory note is a markdown file with three special sections the graph understands:

Frontmatter — YAML at the top: title, tags, permalink. Gives the note a stable identifier even if renamed.
Observations — categorized, atomic facts. Each one is indexed individually. Format: - [category] the fact #optional-tag. The categories that matter: [decision], [fact], [technique], [requirement], [insight], [reversed], [rejected], [open]. Use the full set — they encode not just what's true but what's not true anymore (reversed), what was considered and rejected, and what remains unresolved.
Relations — typed links between notes. Format: - implements [[Other Note]] or - requires [[Other Note]]. The verb before the wiki-link becomes the relation type. Types that matter: implements, requires, part_of, extends, contrasts_with, depends_on, affects. Forward references are fine — linking to a note that doesn't exist yet; Basic Memory resolves the link when you create the target.

What this buys you over plain markdown

Three things a plain folder of notes can't do:

  • Fact-level search. When you or an agent searches for "authentication decisions," Basic Memory can surface the specific observations tagged [decision] across your entire vault, not just the documents that contain the word. Each observation is indexed.
  • Graph traversal. The build_context tool starts from a note and follows relations to gather related material. Ask Claude "give me everything related to Current Canon's auth" and it doesn't just return documents matching "auth" — it returns the auth state file plus everything linked to it via implements, depends_on, requires, as deep as you specify.
  • Temporal awareness. Observations marked [reversed] or [rejected] are not deleted; they become part of the history. An agent reading your state file knows not only what you currently believe but what you no longer believe and why. This is how you avoid the agent re-proposing decisions you've already rejected.

The state file pattern

The single most important file in any Basic Memory project is its state file. One per project. Path convention: [project-slug]/state.md. Structure:

---
title: Current Canon — State
tags: [project, state, currentcanon]
---

# Current Canon

pairs of options; aggregated preferences become a personal

too much information.

## Observations
- [fact] Stack is Next.js on Vercel, Supabase, Stripe test mode
- [decision] Use pairwise-choice as primary vote mechanism #product
- [decision] Supabase RLS for access control rather than app-layer auth
- [decision] Cannot read code; agent explains changes in plain English
- [requirement] Signup requires email confirmation before dashboard access
- [insight] The deeper value is voting-as-mechanism, not media-ranking

  feature bloat, not core to the thesis

  is the right call until 100 paying users

- requires [[Supabase RLS patterns]]
- part_of [[Foolproof projects]]

This file is the single document an agent needs to understand your project. Under 800 words. Reloaded at every session start via Chapter 11's Prompt 2.

The synthesis session — how to produce a state file from scattered context

If you have past conversations with Claude, ChatGPT, and Gemini, you have scattered context. The temptation is to import all of it into Basic Memory. Don't. Raw imports poison the vault with old ideas, abandoned approaches, and thinking you've since revised.

The right move is a synthesis session: a 90–120 minute interview with Claude that produces one canonical state file per project. The prompt to paste:

I need you to help me produce a canonical current-state
document for my project called [PROJECT NAME]. I have
context scattered across past conversations, and my thinking
has evolved — some early decisions were reversed later. I
want a clean markdown file formatted for Basic Memory's
semantic knowledge graph.

Interview me section by section. Ask one question at a time.
Wait for my answer before moving to the next. Do not
summarize or assume — if I'm vague, ask me to be specific.

Produce a file with this structure:

---
title: [PROJECT NAME] — State
tags: [project, state]
---

# [PROJECT NAME]

## Context
[one paragraph in plain language]

## Observations
Use these categories:
- [decision] choices made, dated where remembered
- [requirement] things that must be true
- [technique] specific methods being used
- [fact] objective state right now
- [insight] things learned that generalize
- [reversed] decisions I no longer hold, with reason — CRITICAL
- [rejected] approaches considered and refused, with reason
- [open] genuinely unresolved

Format as: - [category] specific fact #optional-tag

## Relations
Typed links to other notes using [[Wiki-Link]]:
- implements [[Other Note]]
- requires [[Other Note]]
- part_of [[Other Note]]
- contrasts_with [[Other Note]]
- depends_on [[Other Note]]

## Next concrete step
[NOT a list — the single thing I'd do next with 2 hours]

Rules while interviewing me:
- Flag contradictions in my answers; ask which version is current.
- Never fabricate detail. "UNCLEAR — revisit" if I'm not sure.
- Prefer many small observations over few long paragraphs.
- Keep total under 1000 words.
- Present the file for my review. I'll edit before saving.

Start with question 1 now.

After you approve the output, save it with Chapter 11's Prompt 1. From then on, every new agent session starts with that file loaded.

Importing from other agents — the honest truth

Basic Memory has CLI commands: basic-memory import claude conversations and basic-memory import chatgpt. They pull in your past conversation history as notes.

Use them carefully. The output is raw transcripts — useful as a searchable forensic archive, useless as canonical state. If you import, put the imports in a separate Basic Memory project called archive, not your main vault. Your standing-orders note points agents at the main vault for current state; the archive stays available for reference queries but doesn't pollute current-state reasoning.

Synthesis (Claude interviews you, you produce a small state file) beats import (you dump everything in) for active working memory, every time. Import only if you specifically need forensic access to old conversations.

Configuration for day-to-day use

A few practical choices for Basic Memory that matter:

  • Default project mode. If you only have one project, enable default_project_mode=true in the config — it removes the need to pass project to every tool call. If you have several projects, keep it false and always be explicit about which project you mean.
  • Folder organization. One folder per project under the vault root. Keep state.md at the folder root. Sub-folders for decisions, meetings, research, and so on only if they earn their place.
  • The recent_activity tool. At the start of complex work, ask the agent to check recent activity across the vault. This surfaces what you were last working on and prevents re-starting cold.
  • Search type. Basic Memory supports keyword, vector (semantic), and hybrid search. Hybrid is the right default. Vector alone misses exact matches; keyword alone misses rephrased concepts.

What Basic Memory does not do

Same honest naming as every other tool in this book:

  • It does not automatically reconcile contradictions. If you have two observations that disagree, the graph holds both. You resolve.
  • It does not replace your thinking. The synthesis session is you talking through your project out loud in the presence of an interviewer. The interviewer helps; it does not do the thinking.
  • It does not protect against agents writing garbage to it. If Claude writes an observation you disagree with, it's in your vault until you edit it out. Review what agents write to memory weekly (Chapter 16).
  • It does not replace a journal, a diary, or your own writing. See Chapter 4. Personal reflection is not problem-shaped; don't put it here.

How Claude's three personalization layers compose with this

Claude Desktop has three built-in personalization features: Profile Preferences (account-wide), Project Instructions (per-project), and Styles (format/tone). These are UI settings stored in Anthropic's cloud, not files. They compose with standing-orders and AGENTS.md as follows:

Part V Runtime

Three chapters. What happens after the setup is done and the real work begins. How to verify that the agent did what it claimed, how to read the four surfaces of production, and how to keep bridges honest when you have a dozen services that must agree.

Chapter 12Verify gates and patterns

Correspondence at rest is an architecture. Correspondence in motion is a set of gates — specific moments where you stop, look, and confirm before continuing. This chapter names the gates and the patterns that pass through them.

The core pattern

Every action that changes state passes through a gate

The gate has three elements: the agent's declared intent (what it says it will do), the agent's evidence of execution (the raw tool output showing what actually happened), and your verification by looking (a dashboard, a screenshot, a log line).

No gate means no correspondence. A session without gates is an unverified claim.

The four gates

Plan gate. Before any non-trivial action, the agent writes what it will do. In Claude Code, this is plan mode. In Claude Desktop, it's the plan-first prompt (D1 in Chapter 11). You read the plan and approve or revise before any write happens. The work of this gate is almost entirely linguistic — you're checking that the agent's plan matches your intent.

Execution gate. As the agent carries out the plan, raw tool output appears inline. Build logs, test output, migration output, command exit codes. This is the evidence that what the agent claimed to do is what actually happened. The work of this gate is looking for red flags — non-zero exit codes, error lines, unexpected warnings.

Verification gate. After execution, you confirm by looking at the visible surface. Open the dashboard, visit the preview URL, check the database view. The state should match what the plan said it would be. For UI changes, Playwright MCP can provide this by taking a screenshot and reading the accessibility tree; for infrastructure changes, the dashboard is the verification.

Memory gate. At end of session, propose updates to the state file for your review. This is Pólya's Look Back operationalized. The state file is how Phase 4 of this session becomes Phase 1 of the next.

The receipts clause, restated

The execution gate only works if the agent produces receipts. A receipt has three parts:

  1. Final command output, pasted raw. The last 10–30 lines of whatever tool ran — enough to see success markers and absence of errors. Not a summary. The actual bytes.
  2. A dashboard view or URL with a timestamp. Something you can click and see. Stripe event at 14:32 UTC. Vercel deployment abc-123. Sentry issue #4567.
  3. For UI changes: a Playwright screenshot description or attached image. The page, after the change, loaded in a real browser.

Put this in CLAUDE.md. Enforce via hooks if possible. Paste as a reminder (Prompt C1) when stakes are high. "Done" without receipts is a format error.

When to skip gates (rarely)

Gates cost time. For trivial work — answering a question, drafting a paragraph, reading a file to understand it — gates are overhead without benefit. The rule: if the action is purely read-only and does not change any state, skip gates. Anything that changes state goes through all four.

Don't let yourself negotiate. A migration is stateful. A deploy is stateful. A webhook config change is stateful. Anything that touches a third-party service is stateful. These get gates even if you're in a hurry. Especially if you're in a hurry — hurry is when gates save you.

A worked session from plan to memory

Max adds a new feature to Current Canon. Full gate sequence in one session.

  1. Understand (U3 prompt). Max states the unknown, data, condition. Claude restates, asks one clarifying question, gets answered.
  2. Plan gate. Claude enters plan mode. Proposes: new column in profiles table; new RLS policy; one component edit; one API route edit. Max reads, approves.
  3. Execution gate. Claude runs the migration via Supabase MCP. Pastes raw output: migration applied. Commits code. Pushes to preview branch. Pastes vercel --prod=false output with preview URL.
  4. Verification gate. Max opens the preview URL. Logs in. Sees the new field. Claude runs Playwright MCP, takes screenshot, describes what's visible. Both agree: feature works. Max opens Supabase Studio, confirms the new column exists, confirms the RLS policy is visible.
  5. Memory gate (L1 prompt). Claude proposes state file updates: new [fact] for the column, new [decision] for the access pattern, removes the relevant [open]. Max edits, approves. Saves.
  6. Deploy gate (subset of verification gate). Max merges to main. Vercel deploys. Sentry shows no new errors for the first 100 requests. Better Stack uptime check passes. Done.

Six gates in one session. Each under a minute. Total gate overhead: under ten minutes. Total debugging avoided: hours to days.

When a gate fails

Gate failure is always the same pattern: the declared intent, the execution output, and the visible surface disagree. The agent said it would X; the output shows X happened; the dashboard shows Y. Or: the output shows a non-zero exit; the agent claims success. Or: the preview URL doesn't render.

Stop. Do not proceed. Run Prompt Stop (Chapter 11). Investigate before acting further. Most gate failures are recoverable if caught in the gate; none are recoverable if missed in the gate and compounded by the next action.

Chapter 13Production correspondence

Production is four surfaces you should be able to read at a glance, each corresponding to one dimension of what your system is doing. Errors, behavior, uptime, deploys.

The four surfaces

Errors — Sentry. What's broken right now, for whom, how often, in what environment. Grouped by fingerprint so one error repeated 10,000 times is one issue, not 10,000. MCP lets the agent query issues, get live context, generate patches via the Seer AI debugger. The dashboard is where you look first when something feels off.
Behavior — PostHog. What users are actually doing. Feature flags, session replays, funnels, LLM traces if you run AI features. MCP covers flags, insights, web analytics, replay metadata. Replay playback is inherently visual — that part stays in the dashboard.
Uptime — Better Stack. Is the site up? How fast? Response time distribution? Any regional outages? Weekly status emails; public status page for users.
Deploys — Vercel (or your host's equivalent). Build logs, environment snapshot, deployment ID, Instant Rollback. Every deploy has a URL that doesn't change; you can always return to any previous state by ID.

The daily surface — a five-minute morning check

Not a dashboard you need to build. Use what each tool already gives you.

  1. Sentry unresolved issues — any new in the last 24h? Any spike in frequency on old ones?
  2. PostHog — signups and active users trend for last 7 days. Any anomaly?
  3. Better Stack — did any monitor fail in the last 24h?
  4. Vercel — was there a deploy since yesterday? If yes, does Sentry show a post-deploy error spike?
  5. Stripe (if you take payments) — any failed webhooks in last 24h? Events view at the bottom shows a red indicator.

Five minutes. Can be scripted — an agent can do it via MCP and summarize in one paragraph. Do this before any agent work that modifies anything, so you know whether you're starting from a clean production state or compensating for something broken.

The moment of a production incident

Something is broken. A user reported it; a Sentry alert fired; the status page flipped red. You have no need to know everything immediately. You need to know four things, in order:

  1. What broke? Sentry issue with fingerprint, stack trace, affected users. Raw. Not the agent's summary.
  2. When did it start? Compare to the last deploy in Vercel. If the timestamps line up, the deploy is your prime suspect.
  3. How many users? Sentry issue volume. PostHog — did signup or key-action funnel drop?
  4. Can I roll back? Vercel Instant Rollback to the previous deployment is almost always the correct first move if the timestamps implicate a deploy. Data fixes come after.

Rolling back is not admitting defeat. It is restoring the last known-good state while you investigate. A deploy that caused an error is still there after rollback — you can redeploy it or merge a fix; the users get service while you think.

Correspondence across the four surfaces

The power of four tool-specific surfaces is that they corroborate. A deploy at 14:02; an error spike starting at 14:03; uptime dropping at 14:03; a drop in completed signups from 14:03 onward. Four tools agreeing that the deploy caused the incident. High confidence; act fast.

When the surfaces disagree — errors but no deploy, or deploy without any error but a uptime dip — you have a more interesting problem. Investigate before acting. Usually the cause is a third-party bridge (Chapter 14).

Ask Claude for the whole picture

Prompt for a quick production reading:

Give me a 60-second production reading:
- Sentry: any unresolved issues in last 24h, grouped.
- PostHog: signup and primary-action trend, last 7 days.
- Better Stack: any monitor failures in last 24h.
- Vercel: last deploy time and status.

For each, just facts. If any surface is degraded, explain
in one sentence what you see without speculating about cause.
Paste raw data; I'll read it.

Claude composes across all four MCP servers and returns one paragraph. You read it in a minute and either go about your day or dig in.

The invisible git protocol

For frontend work, GitHub becomes silent infrastructure the agent manages. Your review surface is a chat window, a before/after screenshot, and a preview URL. You never read a diff.

Three Vercel CLI commands replace the entire PR cycle:

vercel deploy --yes --no-wait   # preview URL, no commit required
vercel promote <url> --yes      # promote to production
vercel rollback --yes           # instant alias flip, no rebuild

After a one-time vercel link, the agent deploys a preview from any file change, shows you the URL, waits for approval, then promotes. Add the Vercel MCP with claude mcp add --transport http vercel https://mcp.vercel.com so the agent can retrieve deployment status and build logs without leaving the chat.

The verification discipline: the agent takes a before screenshot, makes the edit, takes an after screenshot, and opens both side by side before asking for approval. The before state is not optional — without it, correspondence cannot be checked. Point Playwright at the preview URL, not production. Scope the Supabase MCP to a staging project ref with ?read_only=true until you explicitly authorise write access.

Chapter 14Bridges and supply chain

Your stack has more bridges than tools. Each bridge is a correspondence between two services that must be maintained by you, because neither vendor owns both sides. This chapter names the bridges and the patterns that keep them honest — plus the supply chain below them, which is where attacks silently arrive.

Every bridge is a correspondence

Your signup flow spans Supabase Auth, Postmark (confirmation email), Vercel (the API route that handles the webhook), Stripe (if the user later pays), Sentry (if anything errors), PostHog (if you track the event). That's six services on one user action.

Each hand-off is a bridge. Each bridge can silently fail. The supply chain — the dependencies each of those services depend on, which you don't control — is below all of it.

The three-trait test

A bridge passes the test when its visible surface shows all three of the following. These are the surfaces that make silent failure impossible to hide.

The three-trait test for bridges
  1. Timestamps on every event. You must be able to say with certainty when an event happened to within a second.
  2. Status colors or explicit success/failure markers. Every event is green or red, and red events are visible without scrolling.
  3. Expand-and-replay. You can click into any event, see its full payload, and re-send it. This is the proof that a failed event is recoverable.

If your bridge's visible surface has all three, the bridge is trustworthy. If any is missing, trust is assumed — and assumption is where bridges fail silently.

Stripe's Events view has all three. Supabase's Logs have all three. PostHog's Live Events has the first two; replay is Get-request-only. Postmark's Activity has all three. This is why these tools are Tier S or A — because their bridges to your code are verifiable by looking.

Patterns for the three common bridges

Webhooks (service → your app)

Every webhook should have: a signed secret (HMAC validation so you know the sender is real); idempotency (the same event processed twice doesn't cause double-billing); and the sending service's Events view open as a tab during any testing. The three-trait test applies.

When a webhook fails, the failure is visible in two places: the sending service's Events view (the dashboard shows the delivery attempt, the status, the response from your endpoint); and your app's logs (Sentry if the failure was an exception in your code; your own logs if the handler caught and logged the failure). Both should agree. When they don't, the webhook handler is doing something the sending service can't see.

OAuth (user grants you access)

The token lifecycle is the correspondence. Initial grant shows both in your app (the user is signed in) and in the provider (an active session). Token refresh should be automatic and logged. Token revoke should appear in both places within minutes. Test this quarterly: revoke a token in the provider's dashboard; verify your app sees it gone and handles the loss gracefully.

Scheduled jobs (Inngest, Trigger.dev, n8n)

The job platform has a dashboard; your app has no native view of "did the nightly sync run?" The bridge is the platform's dashboard showing successful runs, and your app's state showing the effect of those runs. Test by tampering: fail a job deliberately; verify your alerting catches it and your app behaves gracefully until the retry succeeds.

The supply chain — four layers of defense

Below every tool is its dependency graph. You don't control it, but you can bound the blast radius when something goes wrong.

Layer 1 — Contain (OrbStack). Run development in containers. A compromised npm package can't reach ~/.ssh from inside a container that doesn't have it mounted. The contain layer puts a wall between your day-to-day work and the dependency graph.
Layer 2 — Audit (Socket.dev). Dependencies get scored for supply-chain risk before they reach your lockfile. Socket catches typosquats, suspicious install scripts, and packages that phone home. Runs automatically on every PR if you set it up that way.
Layer 3 — Block (Little Snitch). Outbound connections from your Mac require approval. A package that somehow bypasses Socket and runs locally still can't exfiltrate data without hitting a prompt. Run in Alert mode for the first few weeks; approvals become routine, anomalies stand out.
Layer 4 — Pin (lockfile discipline). Always commit package-lock.json or bun.lockb or pnpm-lock.yaml. Never delete it. Never regenerate without review. Changes to the lockfile get their own PR, reviewed by a human or an adversarial agent, not folded into a feature PR where they're invisible.

These four compose. Any single layer can fail — and occasionally will. The combination is resilient. A defense-in-depth posture for a non-coder who cannot audit code.

The bridge audit — weekly, 15 minutes

Once a week, prompt Claude to do a correspondence audit across your three most-trafficked bridges:

Run the correspondence audit:

1. Stripe: last 7 days of events. Any failed webhook
   deliveries? If yes, list them with timestamp and
   retry count.
2. Supabase: auth events last 7 days. Any unexpected
   signups, deletions, or role changes?
3. Postmark (or Resend): last 7 days of delivery. Any
   bounce rate above 1%? Any deferral spikes?

Report findings only — don't fix anything. Raw
numbers, no speculation.

Findings go into the relevant project's state file as [open] observations if they need follow-up. Fifteen minutes to a known correspondence baseline for the week.

Part VI Work

One chapter. Eight domains where a solo operator's work actually happens, and the shape of applying the correspondence principle in each. Brief by design — these are patterns, not tutorials. Depth comes from practice, not from reading.

Chapter 15Workflow library

Every domain has the same shape: a problem to understand (Pólya Phase 1), a plan (Phase 2), execution with receipts (Phase 3), and a state-file update (Phase 4). The tools change; the pattern doesn't.

1 · Software work

The canonical domain. Plan mode on. CLAUDE.md at every repo root. Subagents for review. Playwright MCP for UI verification. Git for every change. The receipts clause enforces everything.

One opinionated rule: a session that's touching code should be in Claude Code, not Claude Desktop. The affordances are different — Claude Code has the filesystem and git; Claude Desktop has chat and artifacts. Move to Claude Code when the work turns into editing files.

2 · Written work

Claude Desktop, artifacts on, your Style if you've configured one. Load the state file for whatever you're writing about before you start. Do a Pólya Phase 1 before drafting — what's the unknown (what piece are we trying to produce), what are the data (sources, notes, past drafts), what's the condition (length, audience, tone, non-negotiables).

The mistake to avoid: asking Claude to write the whole piece in one pass. Better: a structured outline, then section-by-section drafts you react to, then a final polish pass with clear instructions. The final piece reads like you wrote it because your reactions shaped every section.

Remember Chapter 4 — creative writing for its own sake isn't problem-shaped. Don't use the agent for that.

3 · Audio and video

Recording in Granola (or equivalent meeting recorder): the transcript is local, yours, and exportable to Basic Memory with a single prompt after the session. For long-form video editing, the agent's role is mostly metadata work — chapter markers, SEO descriptions, transcripts — not the edit itself. Descript is the one agent-native editor worth considering if audio/video is core to your work; otherwise stay in specialized tools and use the agent for the text surrounding them.

For YouTube channels specifically: the pattern that works is to transcribe every video into Basic Memory as it ships. Over a year, you accumulate a searchable corpus of your own thinking that becomes grounding for future scripts and articles.

4 · Research

Claude Desktop, web search on. The Pólya discipline matters most here, because research problems are famously easy to get wrong. State the unknown (what question are you answering, specifically), the data (what you already know, what sources you trust), the condition (what counts as a good answer — one citable source? a systematic review? a decision you can act on today?).

For deep research, use Claude's extended reasoning mode if available; allow the agent to take a longer pass. Verify every non-obvious citation — Claude can hallucinate citations that look real. The receipts clause for research is: paste the source URL; paste the relevant passage; let me verify the passage is actually there before I accept the citation.

5 · Learning

A domain where Chapter 4's warning applies most strongly. Use Claude for: explaining a concept from five different angles until one clicks; practicing translation; drilling flashcards you've already made. Do not use Claude for: writing your essays, solving your problem sets, generating the insights the class is supposed to teach you. The friction is the point.

For a student specifically: keep a learning/state.md with courses and topics. Record the observations and insights from each study session in the relevant topic's subfile. Over a semester, the state file becomes the best study guide you'll ever have, because it's the synthesis you already did.

6 · Administration

Email triage (draft responses Claude writes that you edit and send), calendar scheduling via Cal.com MCP, invoice review, expense tracking. The pattern: the agent does the text work; you do the deciding.

For operators in Germany specifically — and this is where Appendix B matters — the domain is structured by legal requirements. GoBD requires certain documents to be retained for 10 years in unalterable form. DATEV, Lexware, and sevDesk all have APIs; none have MCP at the time of writing, which drops accounting to Tier B. Your Steuerberater (tax advisor) will typically dictate which system you use based on what they accept. The correspondence pattern: the agent drafts entries; you review; the Steuerberater has final say.

7 · Communication

Customer emails, investor updates, marketing drafts. Same as written work but with the added constraint of real human relationships at the other end. Remember Chapter 4: the apology you write yourself lands; the Claude-drafted apology is perceptible. Use the agent for drafts you edit heavily, for research on what's been said before, for holding a long thread of context about a specific customer. Do not use it for the human moment when it arrives.

8 · Daily life

Meal planning with constraints. Trip logistics. Shopping lists. Home inventory. The low-stakes operational work where the agent saves real time without risking much. Keep these separate from your work vault — a life Basic Memory project is fine.

The rule: if removing the agent would damage the experience (cooking with attention, walking in a new city, a meal with a friend), remove the agent. If removing it only costs efficiency, the agent is fine.

The universal shape

Every one of these eight domains is a variation of the same four-phase loop. Understand the specific problem for this domain. Devise a plan using the tools this domain supports. Carry out with the receipts format appropriate to the domain (code receipts for software, citation receipts for research, screenshots for UI work). Look back and update the state file.

The state file is what makes the domains compound. Over a year, your vault has accumulated decisions, reversed decisions, insights, and rejected approaches across every domain you work in. That vault is your operating memory. A new session in any domain loads the relevant state file and starts 10 minutes ahead of where it would otherwise start. Over months, this is the compounding advantage of working in correspondence with an agent.

Part VII Resilience

Three chapters. The cadences that keep correspondence from decaying. The playbooks for when something breaks. The honest map of blindspots that no vendor has yet solved — which matter because they're where your attention has to compensate.

Chapter 16Cadences

Correspondence decays. Not because anyone did anything wrong — because time passes and things change. Rules that used to hold stop holding. Tokens expire. Vendors ship changes. Your own thinking evolves. Cadences are how you catch the decay before it compounds.

Daily — five minutes

The morning check from Chapter 13. Sentry, PostHog, Better Stack, Vercel, Stripe. Anything red or anomalous gets looked at; nothing else. Five minutes.

This is not a ritual. It is the minimum you need to know whether production is in a state where you can safely start agent work. If production is broken, agent work makes it worse. If it's fine, you can proceed.

Weekly — 30 minutes

Sunday evening or Monday morning. A single session in Basic Memory.

  • Backup verification. Restore one random file from yesterday from each of your two backup surfaces (Time Machine and B2). Confirm it opens. Record in a log note: date, file, source, result.
  • Bridge audit. Prompt L4 (Chapter 11). Claude queries your three most-trafficked bridges and reports divergences. Read. Findings become [open] observations in the relevant project state file if follow-up is needed.
  • State file drift check. Open each active project's state file. Is it still accurate? Any observation that's silently become false in the last week? Move stale [decision] entries to [reversed] with a dated reason. Add any new [insight] entries that emerged from the week.
  • Week preview. One prompt: "Given my state files and this week's Linear tickets, what is the single most important thing I should do this week?" Argue with Claude if the answer is wrong. End the session with a written answer.

Monthly — 90 minutes

First Saturday of the month. A longer, more reflective session.

  • Dashboard screenshots. Take a full-screen screenshot of every tool's dashboard — Supabase, Vercel, Stripe, Sentry, PostHog, Cloudflare, Better Stack, Basic Memory. Store in a folder dated by month. Over a year, you have 12 snapshots of the state of your system. This is how you notice cross-provider environment variable drift before it bites you — a named blindspot in Chapter 18.
  • YubiKey check. Actually use each key once. Plug in; tap; confirm it still works. Hardware fails; better to find out on a Saturday than when you're locked out.
  • Access review. What third-party apps have OAuth tokens to your Google, Stripe, GitHub, Supabase? Revoke anything you don't recognize.
  • State file audit via Claude. "Read my most-active state file. Tell me which observations are over 60 days old, still marked as current fact, and have not been touched since. For each, ask me if it's still true." Updates the vault.
  • Envelope check. Physical envelope with recovery codes: still sealed, still where you put it, contents still the right things.

Quarterly — 2 hours

Once a season.

  • Full restore drill. Pick a directory. Delete it. Restore from B2. Diff the restore against git. Any file that's not in git and not restored is a gap.
  • OAuth revoke-and-re-grant. For each critical third-party integration, revoke the OAuth, verify your app detects it, re-grant. Catches token lifecycle bugs that would otherwise hit you at 3am.
  • AGENTS.md and CLAUDE.md review. Read them. Still current? Any rule that's stopped earning its place? Any surprise from the quarter that deserves a new rule?
  • Stack reassessment. For each tool in your stack, is it still Tier S or A for its axis? New contenders? Chapter 5 evaluation test on anything that's changed.

Annual — one day

Once a year.

  • Password rotation for high-value accounts. Email, Bitwarden master, cloud providers. Bitwarden auto-suggests; you just have to do it.
  • Hardware key replacement planning. YubiKeys last years but not forever. After three to five years, budget for a generation of replacements and plan an enrollment day.
  • Stack migration reassessment. Anything moved to Tier D in the last year should be migrated out. Anything new in Tier S should be evaluated for inclusion.
  • Vault cleanup. Archive projects that have gone dormant; merge duplicates; rename files that don't match your current naming convention.
  • This book. The principles don't change. The specific tool assignments do. Re-read Chapter 5; check every recommendation against what's actually shipping today.

Why cadences are enforcement, not willpower

You will not remember to check backups weekly. You will not notice state file drift spontaneously. You will not do the quarterly restore drill because nothing will remind you. This is not a character flaw; it is how human attention works with infrastructure that runs itself correctly most of the time.

Put every cadence on your calendar. Link each calendar event to a note in Basic Memory with a checklist for that cadence. The calendar reminds; the note guides; the entry you type in it confirms. Miss a cadence and the blank row in the log tells you. Over months, the logs are how you know the system is still working as intended and not silently drifting into a state you can't recover from.

Chapter 17When correspondence breaks

Six playbooks for things that will go wrong. None of them is a panic; all of them have a shape. When you're in the moment, you don't want to be figuring out the shape — you want to read it and follow it.

Playbook 1 — You lost a hardware key

  1. Don't panic. You have a backup YubiKey. That's why you enrolled two.
  2. Open your tracking checklist (from Day 2). Every service where the lost key was enrolled.
  3. Log in to each with the backup key.
  4. Remove the lost key from that service's security settings.
  5. If you have spare YubiKeys, enroll a new backup. Order more if you're down to one.
  6. Update the envelope: dated note that the lost key has been revoked from all services, with the date.
  7. If the lost key might be in the hands of someone who knows you, also change the passwords on your critical accounts (email, Bitwarden, cloud providers). The key without the password is useless, but defense in depth.

Playbook 2 — A credential you think is compromised

A password you suspect leaked, an OAuth token you accidentally committed to a public repo, an API key that may have been pasted into a chat with the wrong scope.

  1. Revoke first, investigate second. If you're wrong, the cost is one minute of re-authentication. If you're right, every minute of delay is exposure.
  2. Revoke the specific credential (not the whole account). Passwords rotate in Bitwarden; OAuth tokens revoke in the provider's dashboard; API keys revoke and regenerate in the issuing service.
  3. Audit access logs for the last 30 days. Any unexpected access? Sentry and each provider's audit log.
  4. If anything suspicious in the logs, widen: assume the credential is known, rotate adjacent credentials too.
  5. If the credential was in a public repo: the repo history still has it. Rotate; also purge using git filter-repo or GitHub's secret scanning.
  6. Document the incident in a Basic Memory note: what leaked, how, what was rotated, what was checked, what's still open.

Playbook 3 — The agent did something you didn't intend

A file was deleted. A migration was run. An email was sent. Something is different and you didn't mean for it to be.

  1. Stop. Prompt Stop (Chapter 11). No more agent actions until you understand what happened.
  2. Ask the agent for the full sequence of tool calls it just made, including timestamps and arguments. Not a summary. The list.
  3. Diff against what you expected. The gap is where the surprise happened.
  4. Assess reversibility. Code change: git revert. Database change: point-in-time recovery (Supabase). File deletion: Time Machine or B2. Email: if it's been less than a minute and you're in Gmail, unsend. Otherwise, the email is gone.
  5. If the change is reversible, reverse it. If it's not, accept and mitigate — tell affected users, update Sentry, note the fact.
  6. Update AGENTS.md or CLAUDE.md with the specific rule that would have prevented this. The agent will not make this particular mistake again because you've written it down.

Playbook 4 — A bridge failed silently

A webhook stopped firing two days ago and you just noticed. An OAuth token expired and the background sync has been failing silently. The email confirmation for new signups stopped going out.

  1. Identify the bridge. Which two services, which direction?
  2. Open the sending service's dashboard (Stripe Events, Supabase Auth Logs, Postmark Activity). Filter by the last known good time.
  3. Count the events that should have fired and didn't — or fired and failed. That's your backfill surface.
  4. For events that failed at delivery, most services have a "replay" button. Use it.
  5. For events that didn't fire at all (rare; usually a config issue on the sending side), you may need to backfill manually. Ask the agent to write a script, review the plan, test on one record before running on all.
  6. Fix the root cause of the silence. Token expired? Renew and set up expiration monitoring. Handler was crashing? Fix the crash and add a Sentry alert for it. URL moved? Update the config and add a test.
  7. Add the specific failure to your cadence log as a known-past-failure so you notice faster next time.

Playbook 5 — A dependency was compromised

Socket.dev flags a package you're using as malicious. Or news breaks of a wider supply chain attack; one of your packages is named.

  1. Quarantine first. Stop any running containers that use the affected dependency. OrbStack → Stop → confirm no container is running.
  2. Check your lockfile for the specific version. grep your package-lock.json or equivalent. If you're pinned to a known-good version, you're fine — move to step 4.
  3. If you have a known-compromised version: roll forward to the fix, or roll back to the last known-good version. Update the lockfile. New PR reviewed carefully.
  4. Audit outbound traffic from the time the compromised version was in use. Little Snitch logs, then providers' logs. Any unexpected destination?
  5. Rotate credentials that the compromised dependency might have had access to. Environment variables in Vercel, secrets in Infisical, anything that could have been in the process environment.
  6. Document in a Basic Memory incident note: which package, which version window, what was rotated, what was audited.

Playbook 6 — A backup restore failed

You went to restore something; it didn't come back. The backup you thought was working has not been working.

  1. Immediately: what is your current state? The file or system you were trying to restore is now un-restorable from that source. Don't panic — check your other sources.
  2. Check the other backup source. Time Machine if B2 failed; B2 if Time Machine failed. Different media; the problem is usually in one source, not both.
  3. If both failed: this is serious. Check if the file was ever actually in the backup. Some backup tools silently skip files with permission issues. Review the backup config.
  4. If the file is genuinely gone and both backups failed: this is why you have git for code and the state file for context. Reconstruct from whichever source it actually lived in. Accept the data loss; note it.
  5. Diagnose why the backup failed. Often it's a silent config error (a folder excluded that shouldn't be) or a storage issue (bucket permissions changed). Fix the cause.
  6. Run a verified restore of other files from the failed source to confirm the fix worked. Don't trust the same source until a new successful verified restore happens.
  7. Write a Basic Memory incident note. Next quarter's cadence includes a specific check for this failure mode.

The meta-playbook

Every incident follows the same structure: stop, understand, act, document. When you're in a new kind of failure that none of these playbooks covers, apply the structure. Stop making things worse. Understand the full shape before acting. Act minimally and reversibly. Document so the incident becomes a rule for next time.

You will have new kinds of failures. You will also have the same kinds of failures more than once, because most of them are structural. The playbook adapts by adding rules to AGENTS.md and CLAUDE.md. Over two years, you rarely meet a failure that is genuinely new.

Chapter 18Blindspots and evolution

No stack is complete. No principle covers every case. This chapter names the specific places where correspondence is not yet achievable in 2026 — the gaps your attention has to compensate for, and the products that might close them in the next few years.

Seven named blindspots

Each of these is a real correspondence gap that the recommended stack does not close. Each requires either a workaround, a manual cadence, or accepting the risk.

1. Cross-provider environment variable drift

You have DATABASE_URL in Vercel's environment variables UI, in Infisical, in .env.local on your laptop, and possibly in GitHub Actions secrets for CI. These should all agree. They won't always. There is no tool in 2026 that watches all of these and warns when they drift.

Workaround: monthly dashboard screenshots (Chapter 16). Eyeball the comparison. If you're disciplined about updating Infisical first and running infisical run -- ... locally (never editing .env directly), most of the drift goes away at the source. CI is the most likely to diverge; check it quarterly.

2. Unified OAuth lifecycle

You have OAuth grants to dozens of services — Stripe to Zapier, Google to your calendar app, GitHub to Linear, and so on. Each lives in its own dashboard. There is no unified view of "all third-party apps with access to my accounts."

Workaround: monthly access review (Chapter 16). Check at least the big five: Google, GitHub, Stripe, Supabase, Cloudflare. Revoke what you don't recognize. Products are starting to ship in this space but nothing unified across all providers.

3. Cross-domain agent audit

Claude Desktop's action log shows what Claude did in Claude Desktop. Claude Code's log shows what it did on your laptop. If you also use Codex or Gemini, those are separate logs. There is no unified "what has any agent done on my behalf this week" view.

Workaround: if you can, stick to one agent host (Claude) for most work. If you use multiple, check each one's audit log as a weekly cadence. The state file updates from each session document the intended changes; the agent logs document the actual tool calls. Compare if something feels off.

4. Mac filesystem audit

Every change to your home directory, by any process, is not inventoried. Agents write files; other applications write files; macOS itself writes files. After a month there are thousands of new files and you have no easy way to see which came from where. Spotlight indexes exist but aren't designed for this.

Workaround: contain agent work inside project directories (the working directory scope pattern from Chapter 9). Don't let Claude Code run from your home directory. Git tracks everything inside repos; Time Machine tracks everything; neither is a clean audit surface. Accept that your home directory's non-repo content is the noisy area and keep sensitive work inside repos or an explicitly backed-up folder.

5. End-to-end webhook visibility

A webhook fires at Stripe; lands at Vercel's edge; Vercel routes it to a function; the function writes to Supabase; an Inngest job picks up the change. That's five hops. Each service shows its own view. No single tool shows the trace from Stripe event to final Supabase row, end to end.

Workaround: for critical flows, implement correlation IDs — every webhook generates a UUID, every downstream log line includes it, you can grep across services by ID. Not automatic; requires each service's logging to participate. Partial but tractable.

6. Schema drift between environments

Your local dev database, your preview environments, and production all have their own schemas. Drift happens. Supabase Database Advisors catches some of it (missing indexes, duplicate foreign keys, RLS gaps); Atlas and similar tools catch more (structural differences); none catches all.

Workaround: Supabase Database Advisors on in every project. Migration-only changes to production (never raw SQL). Quarterly schema diff as a cadence. Accept that preview environments may silently drift from production and treat preview-passing as necessary but not sufficient.

7. No correspondence linter

There is no single tool that audits your whole stack for correspondence breaks. No dashboard that says "PostHog is showing 412 signups this week, Supabase shows 418, here are the 6 that are missing." Each correspondence must be audited individually.

Workaround: the weekly bridge audit (Chapter 16) and the correspondence-audit prompt (L4 in Chapter 11) are partial substitutes. Manual but doable in 15 minutes per week. A real correspondence linter is a product that could exist; none has shipped yet.

What to watch for

Three product categories that might close these gaps in the next few years:

MCP Apps extension (shipped January 2026). The January 2026 revision to the MCP spec lets servers declare richer UI affordances. Over the next two years, expect better correspondence between GUI elements and agent actions — click a button on a dashboard, and the MCP-driven agent knows exactly what happened.
Cross-domain audit products. Several early-stage companies are building "agent observability" — a unified log of what any agent did for you across any platform. Whether any of these survive to maturity is unclear; watch the category.
Local-first reactive databases. Projects like Zero, Electric, and the evolution of Convex are pushing toward databases where local state, server state, and agent-visible state are one thing by construction. If these mature, they collapse several of the blindspots above.

The residual risk, stated honestly

Seven named blindspots. Several partial workarounds. No product-level closure of any of them in 2026. This is the honest state of the art for a solo operator working with agents.

The correspondence principle is still the right organizing principle. It does not achieve full closure; it gives you a structure for noticing where closure is missing and for compensating. Your attention closes the last mile. That's not a flaw of the principle — it's what being an operator means.

As the tooling improves, the last mile gets shorter. This book's next edition will have different blindspots; probably fewer. For now, read this chapter when you're feeling overconfident about your setup. These are the places you're not yet protected. Plan accordingly.

Appendix BOperating in Germany

This appendix is written in the register of the book — English — but names the German-specific terms you will encounter. If you operate outside Germany, skip this appendix; if you operate elsewhere in the EU, many concepts transfer but specific agencies, retention periods, and tool vendors do not.

Legal retention — GoBD

The Grundsätze zur ordnungsmäßigen Führung und Aufbewahrung von Büchern, Aufzeichnungen und Unterlagen in elektronischer Form sowie zum Datenzugriff (GoBD) specifies how business records must be retained in electronic form in Germany. Key requirements:

  • Completeness (Vollständigkeit). No business-relevant document may be missing from the archive.
  • Unalterability (Unveränderbarkeit). Once a document enters the archive, it must not be modifiable. This is what Backblaze B2 with Object Lock enforces at the storage layer.
  • Retention period. Ten years for most tax-relevant documents; six for business letters. Starts at end of the calendar year the document was created.
  • Auditor access (Z1/Z2/Z3-Zugriff). The Finanzamt can demand direct read access to your records during an audit.

What this means for the recommended stack: your accounting tool (DATEV, Lexware, sevDesk) handles GoBD-compliant archival of receipts and invoices. Your raw business documents — emails, contracts, project notes — should be retained in a way that preserves them unaltered. Backblaze B2 with Object Lock is a common substrate. Your Steuerberater will specify exactly what goes where.

Accounting — the Tier B reality

DATEV Unternehmen Online, Lexware Office, and sevDesk are the three options most German tax advisors accept. All have APIs; none has a first-party MCP server at the time of writing. This drops accounting to Tier B — the agent can read and write via API, but correspondence with the GUI must be verified manually.

Pattern: the agent drafts entries; you review in the tool's GUI; the Steuerberater has final say. Monthly correspondence audit specifically between your Stripe dashboard and the accounting tool to catch drift.

GDPR and DSGVO

If you run any SaaS in Germany, Datenschutz-Grundverordnung (the German implementation of GDPR) applies. A solo operator with a product that collects any user data needs:

  • Datenschutzerklärung (privacy policy) reviewed by a lawyer familiar with DSGVO.
  • Impressum (legal notice) on every public-facing page.
  • Verarbeitungsverzeichnis (record of processing activities) for your records.
  • AV-contracts (Auftragsverarbeitungsverträge) with every processor you use — Supabase, Vercel, PostHog, Stripe. Each vendor publishes their standard AV contract; you sign it.
  • A lawful basis for every piece of processing (contract, legitimate interest, consent). Most SaaS flows are covered by "contract necessary for service delivery"; analytics usually needs consent.

This is not a domain to handle with only an agent. A Steuerberater and a Fachanwalt für IT-Recht (IT lawyer) are the correspondence you need here. The agent can draft; the professionals sign off.

SIM protection and identity

The German term for a port-out PIN is Kundenkennwort or Schutz-PIN. Call your mobile carrier (Telekom, Vodafone, O2, 1&1) and request one. Without it, a SIM swap attack that compromises any SMS-based 2FA you have remaining is trivial. With it, the attacker needs to social-engineer through your carrier's verification — a real but higher barrier.

Banking

Most German banks have strong 2FA mandated by PSD2 (TAN via app, photoTAN, or similar). API access for solo operators is typically limited — the agent cannot usually drive your banking directly. Accept this as a correspondence break: banking is a high-value domain where manual operation is the default and the agent's role is limited to drafting payment instructions for you to execute.

What the principle looks like for a German solo founder

The rest of this book's recommendations apply with one overlay: every document that touches money or tax goes into the accounting tool's GoBD-compliant archive, and the bridge between that archive and your primary correspondence surfaces (Stripe, Supabase, email) is audited monthly. This is additional work. It's not optional. The upside: when the Finanzamt asks for records in 2034, you have them, unaltered, dated, complete.

Appendix CGlossary

Terms used in this book, in alphabetical order. Brief and operational.

AGENTS.md
A per-repo markdown file at the root of a code repository, giving project-specific standing orders to any agent that honors the AGENTS.md convention (Claude Code, Codex CLI, Gemini CLI). Lives on disk; committed to Git. Read first at session start alongside the cross-project standing-orders note in Basic Memory.
Artifacts
Claude Desktop's mechanism for producing standalone outputs (documents, code, diagrams) that render in the UI rather than appearing as chat text.
Basic Memory
A semantic knowledge graph built on markdown. Supports typed observations and relations. Accessible via MCP for agents; the files remain yours.
Behavior correspondence
A weaker substitute for resource correspondence, used where code connects a visible surface to an agent interface via a build pipeline. Not true correspondence; verification-by-observation (preview URL, Playwright screenshot, subagent review). See Chapter 1.
Bridge
The connection between two services that must stay synchronized (webhook, OAuth, scheduled sync). The place most correspondence breaks happen.
CLAUDE.md
A project-specific agent-instructions file at the root of a code repository. Read by Claude Code at session start.
Correspondence
The property that visible surface and agent interface both reach the same state. The book's central principle.
Finanzamt
German tax authority. Audits business records; can demand electronic access.
GoBD
German regulation specifying how business records must be retained electronically. See Appendix B.
MCP
Model Context Protocol. An open standard for agents to connect to tools. Donated by Anthropic to the Agentic AI Foundation in December 2025.
Object Lock
A cloud-storage feature that prevents objects from being deleted before a retention period expires. Makes ransomware attacks on backups impossible during the lock period.
Observation (Basic Memory)
An atomic, categorized fact inside a note. Format: - [category] fact. Categories used in this book: decision, fact, technique, requirement, insight, reversed, rejected, open.
Overlap
A failure mode where both visible surface and agent interface exist but have diverged. See Chapter 2.
Plan mode
Claude Code's read-only planning mode. Activated by Shift+Tab twice. The agent proposes actions; you approve before execution.
Pólya
Hungarian mathematician (1887–1985). His four-phase problem-solving method (Understand, Devise, Carry Out, Look Back) is the structural foundation of Part IV.
Receipts clause
The agent-instruction rule that "done" without raw output, a dashboard link, and (for UI) a screenshot is a format error, not a completion.
Relation (Basic Memory)
A typed link between notes. Format: - verb [[Other Note]]. Verbs used in this book: implements, requires, part_of, extends, contrasts_with, depends_on, affects.
Resource correspondence
The tight form of correspondence, where a visible surface and agent interface both reach the same state for a resource the system holds (database row, account, deploy, setting, secret). Contrasts with behavior correspondence. See Chapter 1.
RLS
Row Level Security. Postgres/Supabase feature that enforces per-row access control. The correct pattern for multi-tenant SaaS authorization.
Standing orders
A note in Basic Memory holding your cross-project rules — who you are, communication preferences, the tools you operate, the non-negotiables. Read at the start of every agent session. Portable across Claude, Codex, Gemini. Replaces the book's earlier (incorrect) suggestion to put AGENTS.md in Basic Memory.
Steuerberater
German tax advisor. Solo operators in Germany almost always work with one; they specify which accounting tool you use.
Three-trait test
The bridge-evaluation test: timestamps, status colors, expand-and-replay. See Chapter 14.
Tier S / A / B / C / D
The correspondence ranking framework introduced in Chapter 5.
Underlap
A failure mode where a capability exists in only one layer — visible only, or agent-driveable only. See Chapter 2.
Verify gate
A moment in a session where you stop, look at evidence, and confirm before continuing. See Chapter 12.
YubiKey
A hardware authentication device. Required for every sensitive account in this book's setup.

Appendix EColophon

This book was written between March and April 2026 by a solo SaaS founder in Witzenhausen, Germany, in collaboration with an AI agent (Claude) operating under the principles the book itself describes. The principle of correspondence is not original to this book. Variations appear in systems theory, in philosophy of science (the correspondence theory of truth), in accounting (double-entry bookkeeping), and in the lineage of literate programming and infrastructure-as-code.
